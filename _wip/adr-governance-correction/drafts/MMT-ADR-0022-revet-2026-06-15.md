# MMT-ADR-0022 — Activity ledger is the template-first narration and moment substrate

> **Note on numbering:** originally minted as `MMT-ADR-0020` on the new-llm branch (the next free number at the time of authoring); renumbered to `0022` on 2026-06-12 when merging into main — `MMT-ADR-0020` was yielded to the identity-foundation cutover-plan consent-request ADR (analysis C6, WI-678), and `MMT-ADR-0021` was taken by the freeform-threshold ADR (also renumbered from 0019 in the same merge).

**Status:** Accepted · 2026-06-11 · **Scope:** Learner-moment narration & event substrate (the `/now` feed and future visibility / reporting / GDPR-timer surfaces) · **Deciders:** Architect (jjoerg) + PM (owner) + Codex · **Builds on:** MMT-ADR-0000 (decisions layer)

## Context

Several surfaces need a small, deterministic stream of recent durable learner moments — session filing, topic mastery, retention becoming due, recap readiness, snapshot readiness. The first consumer is the learner-home `/now` feed; later visibility / reporting surfaces and GDPR-timer narration (deletion, retention, and consent countdowns) need the same stream. Reconstructing those moments after the fact from scattered operational tables — or re-running LLM summaries — is fragile and non-deterministic: some moments are not cleanly recoverable once the operational state has moved on, and a countdown surface needs a stable row-backed source of "what happened."

The substrate that records them is `mentor_activity_ledger`, an append-only activity ledger.

## Decision

Create `mentor_activity_ledger` as an additive, profile-keyed table and expose a best-effort `writeActivityMoment()` helper. The writer is **non-core**: failures are captured through the existing `safeWrite` posture and never break the primary job or request that produced the moment.

Ledger rows are **template-first**. A row stores a stable `kind`, a stable `templateKey`, and JSON `params`; rendering resolves copy from templates and parameters. The system does not call an LLM per ledger row by default.

**Identity coupling is deferred.** The table is initially keyed by `profile_id`; a later additive migration repoints it from `profile_id → person_id` (FK to `person.id`) and adds the approved relationship-edge reference (nullable FK to `supportership.id`) once the identity model lands. Until then the ledger deliberately neither reads nor writes `person`, `membership`, `guardianship`, or `supportership` — it is delivered and validated independently of the identity cutover.

## Consequences

- `/now` can rank recent moments deterministically with no LLM in the feed path.
- Producers can add narration moments without making non-core writes part of their success criteria.
- The future GDPR-timer surface has an append-only substrate instead of reverse-engineered operational state.
- New ledger kinds remain app-level contract changes, not database migrations, because `kind` and `templateKey` are text validated by `@eduagent/schemas`.
- The identity repoint migration and relationship-edge-reference decision are deferred to a later additive migration; until it lands, all ledger reads remain `profileId`-scoped.

## Alternatives considered

1. **Rank and write feed moments with an LLM.** Rejected. The mentor-home feed must be deterministic, cheap, explainable, and testable; LLM-assisted ranking can only return as a later, separately ruled feature.
2. **Derive moments directly from operational tables every time.** Rejected. Some narration events are not cleanly reconstructable after the fact, and GDPR-timer narration needs a durable event-like substrate.
3. **Key the table by `person_id` immediately.** Rejected. The current pre-identity application still runs on `profiles`; front-loading identity coupling would bind the ledger to an unshipped identity model and block delivering and validating it independently of the identity cutover.
4. **Use a database enum for moment kind / template key.** Rejected. Moment kinds are product-contract changes that should ship additively through schemas and code; a DB enum would force a migration for every new moment without adding meaningful safety.

## Rollout and rollback

The migration is additive-only: it creates `ledger_visibility`, `mentor_activity_ledger`, and two indexes. It does not alter or drop existing objects, so the repo's destructive-migration rollback-section requirement does not apply.

If the ledger ever needs to be removed before it carries user-visible state, forward recovery is `DROP TABLE mentor_activity_ledger; DROP TYPE ledger_visibility;`. Once user-visible moments depend on it, removal requires a separate data-retention decision.

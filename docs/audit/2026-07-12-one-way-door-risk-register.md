# One-Way Door Risk Register - First Pass

**Date:** 2026-07-12
**Scope:** Read-only audit of architecture, ADRs, canon, strategy docs, specs, plans, memory pointers, and selected load-bearing code.
**Status:** First pass. This is a meta-audit record, not product canon.

## Method

This pass treated a "one-way door" as any decision that is hard, costly, risky, or politically painful to reverse because it creates durable lock-in through persisted data, public behavior, irreversible migration, operational dependency, compliance obligation, external contract, app-store/release constraint, or future product strategy commitment.

Inputs inspected included:

- `AGENTS.md`, `CONTEXT.md`, `.claude/memory/MEMORY.md`
- `docs/INDEX.md`, `docs/architecture.md`, `docs/project_context.md`
- `docs/adr/`, `docs/canon/identity/`, `docs/compliance/`, `docs/specs/`, `docs/plans/`, `_wip/`
- selected code under `apps/api`, `apps/mobile`, `packages/schemas`, `packages/database`, and release/config files

## Suggested Next Audit

Run a stricter second pass that starts from the decision-ADR baseline and asks, for each grandfathered `docs/specs` / `docs/plans` decision block:

1. Is this still live?
2. Is it already covered by an ADR/canon line?
3. Is it obsolete and safe to tombstone?
4. Is it a hidden one-way door that needs an ADR or owner?

That pass should produce a smaller backlog of ADR/canon drains, not another broad risk register.

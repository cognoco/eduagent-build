P5 — E2E Speed Work Breakdown

docs/audit/e2e/p5-speed-breakdown.html · 2026-05-14 (revised after Cloudflare API investigation)

Why this exists

The full Playwright suite takes 23 minutes at 1 worker today. The staging API runs on mentomate-api-stg.zwizzly.workers.dev — a Cloudflare-managed *.workers.dev domain with platform-level rate-limiting that users cannot configure. There are no WAF rules, no rate-limit rules, and no Bot Fight Mode on any of our zones (verified via API). Parallel workers trip Cloudflare's built-in protections (BUG-532: 403/429 on bursts). P5 lifts the ceiling by adding a custom domain we control.

Lane composition — at each scenario

Sub-packages

P5a — DECIDED
Custom domain + WAF allowlist (option 1)
front-load (before P1)

The staging worker runs on *.workers.dev where rate-limits are Cloudflare-managed and not configurable. Fix: add a custom domain (e.g. api-test.zwizzly.com) pointing at the staging worker, then add a WAF skip rule for a secret test header on that zone. Playwright points at the custom domain. This is a stepping stone toward option 2 (dedicated test worker) later — all infra carries over.

Steps

  • Add DNS CNAME record on zwizzly.com zone (e.g. api-test.zwizzly.com → staging worker)
  • Bind the custom domain to mentomate-api-stg worker in Cloudflare
  • Add WAF rate-limit skip rule on zwizzly.com zone: if header X-E2E-Bypass matches a secret value, skip rate-limiting
  • Store bypass secret in Doppler stg as PLAYWRIGHT_WAF_BYPASS_SECRET
  • Update playwright.config.ts / test helpers to set the header + point at the custom domain
  • Run full suite at 4 workers, measure wall-clock, confirm no 429s

Exit criteria

  • Full web suite at 4 workers completes in <10 min without 403/429
  • Custom domain + WAF rule documented in canonical runbook
  • Bypass secret stored in Doppler, not hardcoded
Impact: 23m → ~7m for every full suite run. Saves ~16m per iteration on every /goal verify cycle.
P5a+ — PLANNED (later)
Dedicated test worker (option 2)
deferred — after /goal packages complete

Replaces option 1: provision a separate mentomate-api-test worker with its own Neon test DB. Re-point the custom domain from the staging worker to the test worker. Retire the WAF skip rule (not needed — test worker has its own relaxed config). Gives full isolation: test traffic can't pollute staging data, no security risk from the WAF bypass.

What carries over from option 1

  • DNS record (same custom domain, just re-point the binding)
  • Playwright config (same PLAYWRIGHT_API_URL, same value)
  • Doppler secret pattern (swap bypass secret for test-worker secret if needed)

New work

  • New wrangler.toml environment for mentomate-api-test
  • Neon test DB branch/project + DATABASE_URL in Doppler
  • Deploy hook: same code deploys to staging + test simultaneously
  • Remove WAF skip rule from zwizzly.com zone
Impact: full data isolation. No staging pollution from test runs. Clean security posture.
P5b
Tiering — web + mobile lanes
after P1 + M1

Define core, extended, full, manual lanes for both web and mobile. Tag every test. Wire CI: core required on every PR, extended on changed-path PRs, full on nightly + on-demand.

Lane shape

  • core — hard wall-clock cap (5 min web / 10 min mobile). Required on every PR.
  • extended — business-critical journeys. On-demand or path-affected.
  • full — everything except manual. Nightly + on-demand.
  • manual — screenshot crawl, visual review. Off the gate entirely.

Exit criteria

  • No untagged specs/flows (validator-enforced)
  • j01-ux-pass in manual, not core
  • Core lane runtime ≤ 5 min web / ≤ 10 min mobile
  • Lane definitions documented in canonical runbook
Impact: PR-required wall-clock acceptable for stage-2 gating. Full suite still exists, just doesn't gate every commit.
P5c
Mobile parallelism strategy — decision, not code
deferred — decide after P5a/P5b measurement

Mobile Maestro is currently sequential per-flow with ~90s cold bundle. Full regression takes hours. There's no equivalent of "just add more workers." Three real choices:

Options

  • Multi-emulator local — N emulators on a beefy box, manual orchestration. Cheap-ish but operationally fragile, hard in CI without nested KVM / large runners.
  • Maestro Cloud — pay Mobile.dev to fan out. Predictable, ongoing cost. Removes the orchestration burden.
  • Accept sequential + tier aggressively — keep CI mobile suite to ≤10 flows in core, run full nightly on a single emulator overnight. Free, slow, but bounded.

Exit criteria

  • Decision recorded with rationale and cost
  • If Maestro Cloud: budget approved + auth set up + a sample full run completed and timed
  • If accept-sequential: core Maestro lane measured to ≤10 min
Impact: depends on choice. The decision unblocks "what does mobile look like in stage-2 CI gating".

Sequencing — where P5 slots in

infra (P5a) config / decision (P5b, P5c) autonomous /goal runs (benefit from P5a's wall-clock cut)

Risks & mitigations

WAF bypass secret leaks (option 1, temporary)If the X-E2E-Bypass header value leaks, anyone can bypass staging rate limits via the custom domain.
Mitigation: store in Doppler (not hardcoded), rotate quarterly, rotate immediately on leak. Risk retires when option 2 replaces option 1 — the skip rule gets deleted.
Dedicated test worker drifts from staging (option 2, future)Test worker keeps passing while staging API regresses, or vice versa.
Mitigation: deploy hook that pushes the same code+config to both whenever staging deploys. Or: shared deploy job with two targets. Treat divergence as a deploy failure.
Tier creep — "everything is core"Without a cap, teams pile new tests into core "just in case", and the lane bloats.
Mitigation: hard wall-clock cap on core (5 min web / 10 min mobile). CI fails if exceeded. Adding to core requires a justification line in the PR and removing something else.
Maestro Cloud becomes a recurring cost without clear ownershipIf P5c picks Cloud, billing creeps without a budget owner.
Mitigation: approve a monthly budget cap with the decision; route alerts at 80% of cap; tie the choice to "stage 2 mobile gating" objective so the spend has a named purpose.

What this is not

P5 is not a /goal activity. It's a mix of infrastructure (Cloudflare custom domain + WAF rule, later a dedicated worker), CI workflow edits, and a budget decision. Most of the work is human-driven; an agent can help with the code-side config edits but the DNS, WAF, and worker provisioning live outside the repo. P5 runs in parallel with the /goal packages. P5a (custom domain + WAF skip) is decided and ships before P1. P5a+ (dedicated test worker) is planned but deferred until after the /goal packages complete.

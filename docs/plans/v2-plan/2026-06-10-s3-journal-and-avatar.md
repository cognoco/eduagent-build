# S3 — Journal Tab + Avatar Admin Split + Park-and-Return Eval Coverage — Implementation Plan

> **Synced to spec amendment 2026-06-10 (§2.1 mentor character + celebration, §16 voice, §15.17 motivation) and amended 2026-06-13 for earned reward history + learner-authored reflection artifacts.**

> **NO-COHORT RULING — READ FIRST.** The original S3 plan was gated on S1+S2 observed-cohort evidence (spec §11 / §13.6). Product ruled on 2026-06-14 that no S2/S3 cohort evidence data will exist before this work. The gate is removed as an execution blocker rather than treated as passed. S3 proceeds as an explicit product-risk choice behind `MODE_NAV_V2_ENABLED` — **`MODE_NAV_V2_ENABLED` = the V2 three-tab shell build flag; from `EXPO_PUBLIC_ENABLE_MODE_NAV_V2`; reserved API-side at S0 T9, consumed mobile-side at S1; dev/preview/staging-OTA on, prod off**. This ruling does not authorize S6 deletions.

**Goal:** Behind `MODE_NAV_V2_ENABLED`, ship (a) the **Journal tab** — a Me-scope paper-trail of recaps, learner-authored reflections, compact earned-reward history, a **browsable** cross-subject notes archive, and mentor memory, rendering `mentor_activity_ledger` moments — (b) the **avatar admin sheet** that dissolves the More tab and the You-tab hodgepodge by moving settings/language/billing/security/privacy-GDPR/add-child behind the top-corner avatar with owner-gating intact, and (c) **park-and-return eval scenarios in `pnpm eval:llm`** (the **P3 gate** — `P3` = spec §2 principle 3 "park-and-return is the magic"; the one gate that S6's exit-funnel deletion depends on) asserting a parked item surfaces within its window **even under ≤3-slot competition**.
**Approach:** Additive Expo Router screens + a header avatar element behind the V2 flag, consuming the S0 `GET /now`/ledger contracts and the existing recaps/notes/mentor-memory hooks and the reusable `ErrorFallback`/`TimeoutLoader`. The avatar sheet re-homes the *existing* `more/*` screens (no logic rewrite — re-parent + re-gate). Two new eval flows are registered in `apps/api/eval-llm/`: one **deterministic** (no-LLM) flow driving the S0 `buildNowFeed`/`buildNowOverflow` ranking under slot competition, one **LLM** flow driving the conversation-layer re-weaving prose. Nothing the spec assigns to S4 (scope chip, supporter shared-record) or S6 (exit-funnel deletion) is built here — S3 only **adds** the eval coverage that **gates** S6.

## Verified audit amendments (2026-06-13)

These amendments are source-verified and must be resolved before S3 can claim replacement parity.

1. **Me-scope recaps need an API/read-model change.** Current `useRecaps` and `apps/api/src/routes/recaps.ts` are parent-owner/family recap oriented, and the schema does not expose learner reflection/receipt fields. T5 must first add a self-scope recap read model/API (or Journal recap endpoint) for the active profile, include learner reflection/receipt fields, profile-scope the query key, and keep parent recaps owner-gated separately.
2. **Journal memory cannot be read-only if it replaces the self screen.** Existing `mentor-memory.tsx` contains consent, injection toggle, tell-mentor, suppress/delete/restore, and clear-all controls. Journal must preserve those controls or link to the existing management screen until the replacement covers them. S6 must not delete `mentor-memory.tsx` before this parity exists.
3. **`useNowFeed` keys must include active profile.** `/now` uses the active profile header; query keys like `['now','self']` can bleed feed data across profile switches. Use keys such as `['now', activeProfile?.id, 'self']` and `['now', activeProfile?.id, 'overflow', 'self']`, enabled only with an active profile, and add a profile-switch cache-isolation test.
4. **Account avatar/modal mount needs router-layout specificity.** Current tab headers are hidden; adding `account` only to `FULL_SCREEN_ROUTES` hides the tab bar but does not create a modal by itself. S3 must either register account routes in the root stack/modal pattern or add custom V2 top chrome, and must add `account` to the hidden-tab route handling where the whitelist pattern requires it.
5. **Avatar parity includes family breakdown sharing.** Preserve the existing family breakdown sharing toggle/row under the same gates as today, alongside Add child, with owner-with-linked-children visible and non-owner hidden tests.
6. **S1 stub conflict.** S1 creates `app/(app)/journal.tsx`; S3 creates `app/(app)/journal/index.tsx`. These are the same route shape in Expo Router. T8 must replace/remove the S1 stub before adding the folder route and update/remove the S1 stub test.
7. **Eval envelope ruling must be singular.** Ranking flow is deterministic and `emitsEnvelope: false`. Re-weave flow must either use the exchange prompt and return an envelope (`emitsEnvelope: true` with `expectedResponseSchema: llmResponseEnvelopeSchema`) or explicitly stay non-envelope; remove the conflicting `emitsEnvelope` statements before execution.
8. **Deterministic eval hook must fit the current runner.** Current `FlowDefinition` requires `buildPromptInput` and `buildPrompt`, and the runner calls `buildPrompt` before quality/live paths. Either keep those required for deterministic flows with a deterministic notes-only prompt, or change the interface/runner order deliberately. The summary field is `qualityFailures`, not `qualityfailures`.
9. **Me-scope reports need a self-scope read model (same gap as recaps, #1).** The existing report tables `weeklyReports`/`monthlyReports` and their generators are parent/child-pair oriented (`weekly-progress-push`, `monthly-report-cron`); `weekly-self-reports` already writes learner-owned weekly rows to `weeklyReports`. The **My Reports** section (T7b) is **Me-scope only** in S3 — it must read the active learner's *own* reports through a new profile-scoped read model/endpoint (or extend the recaps endpoint), never a parent/child-pair read. The **person-scope** "reports about Emma" rendering (the §6.3 shared record) is **S4/S5** — the same `JournalReportsSection` becomes scope-aware there, behind the S5 reportability gate; S3 builds only the Me-scope list. Monthly reports may not exist for solo learners (the monthly cron is pair-based) — the section renders whatever periodic reports the scope has, with an honest empty state when none.

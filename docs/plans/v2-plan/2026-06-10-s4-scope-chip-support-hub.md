# S4 ‚Äî Scope Chip + Support Hub + Person Scopes + Structural Rendering Mask ‚Äî Implementation Plan

> Synced to spec amendment 2026-06-10 (¬ß3.2 supporter cold-start, ¬ß4.2 pointer module, ¬ß15.18).

**Goal:** Replace today's mode/proxy/tab-shape matrix with a single relationship-lens **scope chip** (Learner = no chip; Supporter = `[Support hub][person]‚Ä¶[Me if studying]`), back the supporter person-scope Subjects view with the **S2 hub component server-masked to structural columns and read LIVE from the supportee's own tables** (never an edge-replicated copy), widen `GET /now` to supporter scopes with a per-edge fairness rule, **consume** the `mentor_activity_ledger` repoint (`profileId ‚Üí personId` + `edgeId`) that the identity-foundation cutover plan's **M-REPOINT** migration owns and lands (S4 does not migrate the ledger column itself), render the **supporter cold-start** surface (variant-zero add-child + the per-child account-state branch: Handoff / Approve / Kickstart + the stale-idle de-escalation arc ‚Äî ¬ß3.2), emit a compact **cross-scope pointer module** into the Me feed so a dual-role adult never misses a struggling child (¬ß4.2), and retire the ModeSwitcher / proxy mode / tab-shape matrix behind the V2 flag while V0/V1 must not regress.

**Approach:** This is the **first phase that reads the identity-foundation person/edge graph** (¬ß9 contract: no S0‚ÄìS3 deliverable touches `person`/`supportership`/`guardianship`; S4 is where it begins). It builds (a) a server-side **scope-resolution service** that enumerates a Person's scopes from their `supportership` edges, (b) a **structural permission mask** that re-uses the S2 hub's read shapes but server-filters to the "grades layer" columns and live-reads the supportee's tables (never copies them), (c) the `/now` supporter-scope widening with per-edge fairness, (d) the ledger writer/reader updates that **assume the `profileId ‚Üí personId` + `edgeId` repoint is already done** by the identity-foundation cutover plan's M-REPOINT migration (S4 does **not** author or generate that schema migration ‚Äî it is external, owned by the IF plan), (e) the mobile scope chip + a V2-flag-gated retirement of the ModeSwitcher/proxy/tab-matrix, (f) the **supporter cold-start surface** ‚Äî the empty-state instances of the hub's per-person modules: variant-zero "+ Add my child" with positive-only door copy + ghost preview, the per-child morphing card that branches on the child's deterministic account state (Handoff / Approve / Kickstart, the branch read server-side for free from the ¬ß6.2 tier ‚Äî no LLM, no topology bet), and the granted-idle **stale-idle de-escalation arc** by visit count, and (g) a compact **cross-scope pointer module** the server emits into the Me feed ("1 thing needs you in the Support hub") that links into the hub without duplicating its attention items. Every artifact-wall guarantee (¬ß6.1) is enforced server-side with negative-path access tests, never client-side hiding. Out of scope: the linking ceremony, two-way transparency views, managed/credentialized tiers, graduation, and the revocation FLOW (all S5); cutover/deletion (S6). The supporter cold-start UI is the S4 Support-hub surface; the **linking ceremony**, **managed-tier activation** (Handoff's day-one gate), and the **consent-pending Approve flow** that the cold-start anchors *land in* are S5 ‚Äî S4 renders the anchors and routes into those S5 surfaces, it does not build them.

**Visibility derivation rule (load-bearing, inv 8 / inv 9):** supporter visibility into another Person's learning data is **edge-derived from `supportership` ONLY**, edge-scoped to the named supportee, never org-wide and never from `membership` or `guardianship`. The structural mask (T7/T8) asserts an **active `supportership` row** (`revoked_at IS NULL`) from the requesting supporter to the target supportee before any read. A guardian who is NOT also a supporter has consent authority but **no everyday data-visibility scope in the chip** (a guardian view may be *derived* per `MMT-ADR-0008` `op(G,C)` but that is the guardian-operate path, distinct from the supporter person-scope this plan builds ‚Äî see OPEN ITEM 2).

### S4 OPEN ITEMS (canon underspecifies ‚Äî record, do not invent)

1. **`edgeId` has no canon column.** Spec ¬ß8.2 calls for the ledger repoint to "add `edgeId`" so a row can be attributed to the edge that produced it (e.g. a supporter-visible moment) ‚Äî that column is added by the **IF cutover plan's M-REPOINT migration**, not by an S4 migration (the repoint is owned by the IF plan; see the Blocked-by ownership note). **The identity canon defines no surrogate `edge_id`** ‚Äî `supportership` and `guardianship` each have their own `id UUID PRIMARY KEY` (data-model.md ¬ß2A key convention: every surrogate PK is `uuid` v7), but there is no single polymorphic "edge" id. **Decision for this plan (recorded, NOT invented silently):** `mentor_activity_ledger.edge_id UUID` is **nullable** and, when set, references **`supportership.id`** (the only edge that carries everyday data-visibility ‚Äî inv 9). It is NULL for self-scope (`visibility='self'`) rows. A guardianship-attributed moment, if ever needed, is a future additive column (`guardianship_edge_id`) ‚Äî NOT overloaded onto `edge_id`. **This polymorphism question is an OPEN ITEM** S4 must raise to the identity-foundation owner so it is settled **before M-REPOINT lands the `edge_id` column**: confirm whether a single typed `edge_id` + `edge_kind` discriminator is preferred over the supportership-only FK.
2. **Guardian-operate vs supporter person-scope.** The canon's `op(G, C)` derivation (`MMT-ADR-0008`: `guardian-link ‚àß shared-org ‚àß charge-has-no-Login`) gives a guardian an *operate* capability over a managed charge ‚Äî distinct from the *supporter* person-scope this plan builds (edge-scoped visibility, any age). **For S4, the chip person-scope is `supportership`-derived ONLY.** Whether a managed charge's guardian also gets a person-scope in the chip (via the derived guardian view) is an **S4 OPEN ITEM** for the identity owner; the launch tier is credentialized (managed activation deferred per ¬ß13.5), so the supportership-only path covers the live audience and the guardian-derived path can be added in S5 when managed activates. Default for S4: do not surface a guardian-only (non-supporter) person-scope.
3. **`displayName` source for a supportee.** The chip shows a person label ("Emma"). Canon `person.display_name NOT NULL` (data-model.md ¬ß4.1). The mask returns the supportee's `display_name` as the **only** identity field crossing the edge for the chip label ‚Äî confirmed structural, not an artifact. No OPEN ITEM; recorded for completeness.

  export const scopeKindSchema = z.enum(['me', 'supporter-hub', 'person']);
  export type ScopeKind = z.infer<typeof scopeKindSchema>;

  // A single chip entry. `personId` is present iff kind === 'person'.
  export const scopeDescriptorSchema = z
    .object({
      kind: scopeKindSchema,
      personId: z.string().uuid().optional(), // the supportee's person.id (kind='person' only)
      edgeId: z.string().uuid().optional(),   // supportership.id backing this person-scope (kind='person' only)

    .refine((s) => (s.kind === 'person') === (s.personId !== undefined), {
      message: 'personId is required iff kind is person',
    })
    .refine((s) => (s.kind === 'person') === (s.edgeId !== undefined), {
      message: 'edgeId is required iff kind is person',
    });
  export type ScopeDescriptor = z.infer<typeof scopeDescriptorSchema>;

  // The ordered chip list. Supporter shape: [supporter-hub, person‚Ä¶, me?]; Learner shape: [].

  export type SupporterScopeList = z.infer<typeof supporterScopeListSchema>;

# Slice 1.5 PR 1.5a — Topic-Probe-While-Teach Prompt + Subject-Level Opener + Signal Reflection

**Date:** 2026-05-07
**Status:** Draft plan, ready to implement
**Branch:** TBD (parallel to Slice 1 work)
**Parent plan:** 2026-05-07 conversation. Net-new slice introduced because the audit's per-subject interview was identified as wrong-grain ("Biology generally" doesn't help teach photosynthesis); the valuable diagnostic is per-topic, woven into teaching, not a separate intake screen.
**Wave:** Slice 1.5 Wave 1 — sequential predecessor of 1.5b and 1.5c.
**Size:** M

---

## Goal

Replace the per-subject `interview.tsx` 3-turn intake with a **per-topic teach-while-probe** mechanic that runs as the first 3-4 turns of any first-encounter topic session. Reuses the existing interview's signal-extraction prompt structure ("react + follow-up question") but reframes it as natural tutor curiosity at the right grain. No new screen, no new route, no step indicator — rides on the existing session screen.

Adds two adjacent surfaces to preserve the "feeling seen" moment that today's interview provides:
1. **Subject-level opener** — one casual question on the very first session of a new subject ("Quick — before I dive in: what brought you to Italian? Anything specific you want to be able to do?"). Captures subject-level goals once.
2. **Signal-reflection rule** — when the LLM extracts a substantive signal from a probe answer, the next turn explicitly references it ("You mentioned you've already played with chemistry sets — let's pick up from there"). Makes the "I'm being seen" moment perceptible.

## Current state (verified 2026-05-07)

### What's already in place to reuse

- **Signal extraction logic** — `apps/api/src/services/interview.ts → extractSignals()` (called from `apps/api/src/routes/interview.ts:477`). Extracts `goals`, `experienceLevel`, `currentKnowledge`, `interests`, `interestContext`, `analogyFraming`, `paceHint` via Gemini Flash from a transcript. Reusable as-is against any transcript shape.
- **Prompt structure** — `INTERVIEW_SYSTEM_PROMPT` in `apps/api/src/services/interview-prompts.ts:8-29`. Already says: "On each FOLLOW-UP reply: briefly react to their answer (confirm, correct, or add one new piece of info), then ask one focused follow-up question." That's exactly the teach-while-probe pattern; just needs reframing and re-targeting.
- **5b first-turn rule** — `apps/api/src/services/exchange-prompts.ts:455-468` (after 5b's edit, commit `a64d9ed2`). First learning response must teach exactly one concrete idea AND end with exactly one learner action. This rule needs a carve-out for first-encounter topics: "teach exactly one concrete idea AND end with one focused follow-up question about prior knowledge or a gap" instead of an action.
- **Non-answer detection** — `apps/api/src/services/session/review-calibration.ts:79-101` (`isSubstantiveCalibrationAnswer`). Locale-aware "idk" / "nei" / "わからない" filter. Reusable for the topic probe (substantive vs. non-substantive answer routing).
- **First-encounter signal** — `apps/api/src/services/retention-data.ts:166-208` (`ensureRetentionCard`). Returns `{ card, isNew }`. The "have I taught this topic before to this learner" boolean is one threading change away from the prompt context.
- **Session context plumbing** — `apps/api/src/services/session/session-context-builders.ts` (referenced from audit § J at line 324) builds the `ExchangeContext` consumed by `exchange-prompts.ts:buildSystemPrompt`.

### What's missing (net-new for this PR)

- An `isFirstEncounter` boolean on `ExchangeContext` (`exchange-prompts.ts:271-282` consumes the context but has no first-encounter awareness).
- A new prompt block in `exchange-prompts.ts` that fires when `isFirstEncounter && exchangeCount < 4`. Block contents: teach-while-probe rule, follow-up structure (carry the existing `INTERVIEW_SYSTEM_PROMPT` paragraph but reframed), "switch to normal teaching once you have enough signal — by turn 4 at latest."
- The 5b carve-out — first-encounter topics replace the "exactly one learner action" line with "exactly one focused follow-up question about prior knowledge."
- A subject-level opener block fired when this is the first session for a never-seen subject (separate signal from first-encounter-topic; only fires once per subject, on turn 0 of session 1).
- A signal-reflection rule — when an extracted signal is in scope (delivered via context), instruct the LLM to reference it on the next turn.
- Eval-harness scenarios under `apps/api/eval-llm/` covering: first-encounter topic turn 0/1/2/3, first-session-of-subject turn 0, returning-topic turn 0 (must hit normal 5b rule, not the probe).

### What's intentionally NOT in this PR

- Deletion of `interview.tsx`. That is Slice 1.5c, gated on this PR shipping AND being verified in production-like staging.
- Wiring the substantive answer into an Inngest event for retention-card seeding. That is Slice 1.5c's ride-along.
- Mentor-memory entry-point cue ("Here's what your tutor knows about you"). That is 1.5b.

## Implementation steps

1. **Add context fields to `ExchangeContext`.**
   - `isFirstEncounter: boolean` — true when no prior `learning_sessions` row exists for `(profileId, topicId)`. Cheap query; can piggyback on existing retention-card lookup if the session-context builder already reads it.
   - `isFirstSessionOfSubject: boolean` — true when no prior `learning_sessions` row exists for `(profileId, subjectId)`. One COUNT query.
   - `extractedSignalsToReflect: { goals?: string; currentKnowledge?: string; interests?: string[] } | null` — populated by 1.5c's async extraction; this PR just reads-or-null. For now, always null — the prompt block guards on null and skips the reflection sentence.
2. **Build the topic-probe block.** Inserted into `exchange-prompts.ts:buildSystemPrompt()` after the first-turn-rule block:
   ```
   FIRST-ENCOUNTER TOPIC RULE: For the first 3-4 turns on a topic the learner
   hasn't seen before, weave one teaching nugget AND one focused follow-up
   question into each reply. The follow-up should react to what the learner
   just said (confirm, correct, or add one new piece of info), then ask about
   a knowledge gap or goal you spotted. Track what they know, what they
   don't, and what they want to do with it. Switch to normal teaching once
   you have enough signal — by turn 4 at latest. NEVER frame this as an
   interview, intake, or assessment. Just be a curious tutor.
   ```
   Trigger: `isFirstEncounter && exchangeCount < 4 && !isLanguageMode && !isRecitation && !isReviewMode`. Language and review have their own existing first-turn blocks; do not double up.
3. **Build the subject-opener block.** Fired only when `isFirstSessionOfSubject && exchangeCount === 0`:
   ```
   SUBJECT OPENER: Before any teaching, open with one casual question:
   "Quick — before I dive in: what brought you to <subject_name>? Anything
   specific you want to be able to do?" Keep it conversational, not formal.
   Only fire on the very first turn of the very first session of the
   subject — never repeat. After their answer, follow the FIRST-ENCOUNTER
   TOPIC RULE above on subsequent turns.
   ```
   Trigger logic: when both `isFirstSessionOfSubject && exchangeCount === 0` are true, the subject-opener block takes precedence over the topic-probe block on turn 0. Turn 1 onward in that same session, the topic-probe block is in charge.
4. **Carve out 5b.** The current 5b rule at `exchange-prompts.ts:455-468` reads "first learning response must teach exactly one concrete idea AND end with exactly one learner action." Modify to:
   - If `isFirstSessionOfSubject && exchangeCount === 0` → subject opener applies (no 5b action requirement).
   - Else if `isFirstEncounter && exchangeCount === 0` → first-encounter topic probe applies (teach + follow-up question, not teach + action).
   - Else → original 5b rule (teach + action).
5. **Build the signal-reflection block.** Fired when `extractedSignalsToReflect != null && exchangeCount > 0`:
   ```
   SIGNAL REFLECTION: The previous turn extracted these signals from the
   learner: <serialised_signals>. Reference one of them naturally in your
   reply ("You mentioned you've already played with chemistry sets — let's
   pick up from there"). Do not list signals robotically; weave one in.
   ```
   This block sits dormant for this PR (signals always null) until 1.5c populates it.
6. **Add tests for `exchange-prompts.test.ts`.** Coverage:
   - `isFirstSessionOfSubject && exchangeCount === 0` produces a system prompt with the SUBJECT OPENER block, no FIRST-ENCOUNTER TOPIC RULE conflicting.
   - `isFirstEncounter && exchangeCount === 0` (not first session of subject) produces the FIRST-ENCOUNTER TOPIC RULE block and modifies the 5b rule to "teach + follow-up question."
   - `isFirstEncounter && exchangeCount === 3` still has the FIRST-ENCOUNTER TOPIC RULE; `exchangeCount === 4` does not.
   - Returning topic (`!isFirstEncounter`) produces the original unmodified 5b rule.
   - Language and review modes are unaffected by the new blocks.
7. **Eval-harness scenarios.** Add to `apps/api/eval-llm/scenarios/`:
   - First-encounter non-language topic (e.g., "Photosynthesis"), turns 0-3.
   - First-session-of-subject, turn 0 (asserts the SUBJECT OPENER framing).
   - Returning topic, turn 0 (asserts the original 5b rule still applies — regression guard).
   - Update existing scenarios that asserted the unconditional 5b rule with `isFirstEncounter: false` so they still represent the "common case" (most session turns are not first-encounter).
8. **Run eval harness.** `pnpm eval:llm` (Tier 1 snapshot). Then `pnpm eval:llm --live` (Tier 2, real LLM call) — confirm:
   - First-encounter topic turn 0 ends with a question, not an action.
   - Subject opener turn 0 references the subject by name and asks one casual question.
   - Returning topic turn 0 still ends with a learner action (5b regression hold).
   - LLM does not call its own behaviour an "interview" or "intake" anywhere.

## Failure Modes

| State | Trigger | User sees | Recovery |
|---|---|---|---|
| LLM still ends first-encounter turn 0 with an action instead of a question | Prompt rule unclear or eval scenario miscategorized | Eval-harness Tier 2 fails | Tighten prompt wording; do not weaken eval assertion |
| Subject-opener fires on subject #2's first session | `isFirstSessionOfSubject` query bug — checks profile-level instead of per-subject | User gets "what brought you here" twice | Fix the query to scope per-subject; add a regression test |
| Topic-probe still firing on turn 5+ | Off-by-one in `exchangeCount < 4` | Tutor keeps probing instead of teaching | Confirm threshold logic; add a turn-5 test |
| Returning-topic regression — original 5b rule no longer fires | Carve-out logic too broad | Every session opens with a probe | Eval scenario for returning-topic catches this; rollback prompt edit |
| Signal-reflection fires on a generic / unhelpful signal | LLM extraction noise | Tutor mentions something the user didn't really say | This PR keeps signal-reflection dormant (`null`); 1.5c addresses signal quality |
| `isFirstEncounter` query slow | Per-turn DB query | First-turn latency spike | Cache result on session creation; never re-query within a single session |

## Risk and rollback

- **Blast radius:** medium. Prompt-only changes to a load-bearing prompt-construction function; affects every learning session.
- **Rollback:** revert the PR. Prompt reverts to pre-1.5a behavior; eval baseline reverts. No data migration involved.
- **Cannot break:** session creation, session streaming, signal-extraction at the existing per-subject interview path (still in `interview.tsx`, not touched by this PR), retention card mechanics, review-mode prompt block.
- **What needs vigilance during review:** that the trigger conditions don't double-fire — subject-opener and topic-probe are mutually exclusive on turn 0 of session 1 of a new subject; the spec says subject-opener wins. Eval scenarios catch this.

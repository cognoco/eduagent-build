# Chat-Only App Help In Mentor Chat

**Date:** 2026-05-13
**Status:** Spec - not yet implemented
**Size:** S
**Source:** Product discussion 2026-05-13

## Goal

Let the mentor answer normal end-user questions about how to use the app from inside the existing chat.

Examples:

- "Where do I find my notes?"
- "How do I change learning preferences?"
- "What is Explorer vs Challenge mode?"
- "Where can I see what the mentor remembers?"

The answer is plain chat text only. No buttons, no deep links, no app-navigation action schema in this slice.

## Why this matters

The mentor currently knows how to teach, but not enough about the app it lives in. That creates a small but sharp trust gap: users can ask a reasonable in-context question and get either a generic answer or a learning-focused redirect.

The desired behavior is simple:

> User: "Where are my notes?"
> Mentor: "Open Library, choose the subject or book, then open the topic. Notes appear in the Notes section there."

This should feel like the mentor knows the room, without turning the learning chat into a support bot.

## Non-goals

- No clickable buttons below messages.
- No arbitrary URLs, internal Expo route strings, or markdown links generated by the LLM.
- No new Help tab or support chat surface.
- No schema change to the LLM response envelope.
- No database migration.
- No exemption from quota/billing in this first slice.
- No live screen awareness. The mentor can describe stable app structure, not inspect the user's current screen.

## Product posture

App-help answers should be:

- Brief: one or two sentences unless the user asks for detail.
- Concrete: use visible labels the user can tap, such as "More", "Preferences", "Library", "Mentor memory".
- Plain-language: no internal terms like route, stack, retention, envelope, or curriculum unless the visible UI label uses that word.
- Bounded: if the mentor does not know a destination, say so and suggest "More > Help & Feedback" instead of inventing.
- Learning-aware: after answering, gently return to the learning thread only when there was one.

Example return-to-learning line:

> "Then come back here and we can keep going."

Avoid this when the whole conversation is app help.

## App Help Map

Create one small server-owned app help map. It should describe stable user-facing destinations, not implementation routes.

Initial map:

| User asks about | Chat answer should point to |
|---|---|
| Notes | "Library > choose the subject/book/topic > Notes section." Mention Library search can find notes if relevant. |
| Saved explanations / bookmarks | "Progress > Saved." Distinguish this from notes when the user says "saved". |
| Learning preferences | "More > Preferences." |
| Learning accommodation | "More > Preferences > Learning accommodation." |
| Explorer mode | Relaxed, flexible learning. The mentor is more encouraging, and the learner can move at their own pace. |
| Challenge mode | More focused learning. The mentor keeps the learner on track and asks for stronger proof of understanding. |
| Changing Explorer / Challenge | In a session, tap the mode chip in the header. Outside a session, use "More > Preferences" if that row is available. |
| Mentor memory / what the mentor knows | "More > Mentor memory." |
| Mentor language / profile | "More > Mentor language" or "More > Profile." |
| Notifications | "More > Notifications." |
| Privacy / data export / account deletion | "More > Privacy & data." |
| Help / feedback | "More > Help & Feedback." |
| Homework | "Home > I have homework" or the Homework entry. |
| Practice / reviews | "Practice" if visible from the app shell; otherwise "Home" entry cards. |
| Parent child settings | "Home > open the child's profile" for child-specific preferences. |

The map should be easy to edit as IA changes. If a destination moves, updating this single map should update prompt behavior.

## Implementation

### Server prompt map

Add a small prompt helper, for example:

`apps/api/src/services/app-help-map.ts`

Responsibilities:

- Export `buildAppHelpPromptBlock(): string`.
- Contain only server-authored static text.
- Use visible UI labels only.
- Include a version/date comment so stale-map updates are easy to audit.

Then inject this block into `buildSystemPrompt()` in:

`apps/api/src/services/exchange-prompts.ts`

Placement:

1. After safety and anti-fabrication.
2. Before topic-specific teaching guidance.
3. Before the final JSON envelope instruction.

Prompt rule:

```text
APP HELP:
If the learner asks how to find, change, or understand something in the app, answer from this map in plain chat text. Do not invent screens, buttons, routes, links, or capabilities. Use visible labels only. Keep the answer short, then return to the learning thread if appropriate.
```

### No envelope change

Keep using the current envelope:

```json
{
  "reply": "...",
  "signals": { ... },
  "ui_hints": { ... }
}
```

For app-help turns:

- `reply` contains the help answer.
- `signals.partial_progress` should be false.
- `signals.needs_deepening` should be false.
- `signals.understanding_check` should be false unless the mentor genuinely asks a learning check after the help answer.
- `ui_hints.note_prompt.show` should be false.

No new `ui_hints.navigation` field in this slice.

### Accounting and learning state

Phase 1 accepts that app-help questions are normal chat exchanges:

- They count as a chat exchange.
- They consume the same quota as any LLM turn.
- They remain in the transcript.

The prompt should prevent them from mutating learning-state signals, but this slice does not add a separate `app_help` event type or billing exemption. If support questions become common enough to pollute learning analytics, add that as a separate feature.

### Mobile

No mobile UI work is required for the first slice.

Existing chat rendering is sufficient because the answer is ordinary mentor text. The LLM must not output markdown links or route-like strings that imply a tappable action.

## Acceptance Criteria

1. A user can ask "Where do I find my notes?" in a normal chat and receive a concise answer pointing to Library and the Notes section.
2. A user can ask "How do I change learning preferences?" and receive a concise answer pointing to More > Preferences.
3. A user can ask "What is Explorer vs Challenge mode?" and receive a concise, user-facing explanation of both modes.
4. The mentor does not invent buttons, routes, URLs, or links.
5. The mentor does not refuse app-navigation questions as off-topic.
6. App-help answers do not set note prompts, partial-progress, or needs-deepening signals.
7. Existing teaching behavior remains unchanged for normal learning questions.

## Tests

### Unit tests

Add tests in `apps/api/src/services/exchange-prompts.test.ts`:

- `buildSystemPrompt` includes the app-help block.
- The app-help block contains the expected destinations for Notes, Preferences, Explorer, Challenge, and Mentor memory.
- The app-help block does not contain internal Expo route strings like `/(app)`.
- The response-format instruction remains last.

Add tests for the helper if split into `app-help-map.ts`:

- It returns stable visible labels.
- It contains no markdown links.
- It contains no raw route paths.

### Eval harness

Update the exchanges eval flow with app-help probes:

- "Where do I find my notes?"
- "Where do I change learning preferences?"
- "What's the difference between Explorer and Challenge mode?"
- "Where can I see what you remember about me?"

Run:

```bash
pnpm eval:llm -- --flow exchanges
```

Because this changes an LLM prompt, commit the resulting prompt snapshots. Run live eval only if the snapshot review shows ambiguity or the model tends to invent navigation:

```bash
pnpm eval:llm -- --flow exchanges --live
```

### Focused API/service validation

Run the relevant prompt and exchange tests:

```bash
pnpm exec jest apps/api/src/services/exchange-prompts.test.ts apps/api/src/services/exchanges.test.ts --runInBand
```

If prompt changes touch broader exchange behavior, also run:

```bash
pnpm exec nx run api:test -- --runInBand
```

## Failure Modes

| State | Trigger | User sees | Recovery |
|---|---|---|---|
| Stale app map | A screen is renamed or moved without updating the prompt map | Mentor gives old directions | Update `app-help-map.ts`; add/adjust prompt-map test |
| Hallucinated destination | Model answers beyond the map | User gets nonexistent screen/button | Prompt says use map only; eval probes catch common cases |
| Internal route leaks | Model says `/(app)/more/learning-preferences` | Confusing technical text | Tests assert the map has no raw routes; prompt forbids route strings |
| App-help counted as learning progress | User asks app question mid-session | Exchange count increments, but learning signals stay false | Accepted for phase 1; add `app_help` event type later if analytics pollution appears |
| Overlong support answer | Model explains the whole IA | Chat feels heavy | Prompt says one or two sentences; eval snapshots review tone |
| User asks for current-screen state | "Where am I now?" | Mentor may not know | Prompt says it cannot inspect the current screen; suggest visible top-level tabs |
| User asks for account/security action | Delete/export/privacy question | Wrong or unsafe advice would be high-friction | Map points only to More > Privacy & data; do not provide policy/legal claims |
| Parent/child context ambiguity | Parent asks how to change child preferences | Mentor gives self-profile directions | Map includes parent path: Home > child profile for child-specific preferences |

## Sequencing

1. Add `app-help-map.ts`.
2. Inject the app-help block into `buildSystemPrompt()`.
3. Add prompt-map unit tests.
4. Add app-help probes to the exchanges eval harness.
5. Run focused tests and `pnpm eval:llm -- --flow exchanges`.
6. Review generated snapshots for tone, route hallucination, and signal defaults.

## Rollback

Safe to revert as a prompt-only/server-only change:

- No schema changes.
- No mobile UI changes.
- No migration.
- No response contract changes.

Rollback is removing the prompt block import/injection and deleting the helper/tests/snapshots added for this feature.

## Future Enhancements

Only consider these after chat-only behavior proves useful:

- Render safe buttons for a whitelist of destinations.
- Add an `app_help` structured signal or event type.
- Do not count app-help turns toward learning exchange metrics.
- Add localized app-help maps if UI labels diverge meaningfully by locale.

# Parent Visibility & Privacy

**Status:** Design spec ŌĆö pending review
**Author:** Product + Claude brainstorm, 2026-04-18
**Depends on:** Nothing ŌĆö can be implemented independently of RLS Phase 0-1
**Companion spec:** `2026-04-18-progress-empty-states-highlights-design.md` (independent, parallel track)

## Problem

The parent dashboard (`/(app)/dashboard`, 10 API routes, 8 mobile screens) was built feature-first: every data type the child generates is exposed to the parent. This includes raw session transcripts (turn-by-turn exchanges with the AI mentor), full mentor-memory inference items, and escalation rung metadata. No privacy policy was defined before the routes were built.

Three problems need resolution before production:

1. **No privacy boundary** ŌĆö parents see raw transcripts of everything the child said to the AI. For a 14-year-old, that's invasive. There's no product decision about what's appropriate vs. what's technically queryable.
2. **RLS gap on `family_links`** ŌĆö RLS is enabled (migration `0027`) but zero policies exist. The application-layer `assertParentAccess` check is the only guard. A service bug or future direct-DB query path bypasses it entirely.
3. **RLS Phase 2-4 plan doesn't account for parent access** ŌĆö the enforcement plan writes `profile_isolation` policies on all child-data tables (`profile_id = current_profile_id`). Once enforced, the parent dashboard breaks because it reads child data using the child's profile_id after an application-layer family-link check. The plan needs parent-access subquery policies on tables the dashboard reads.

## Privacy Policy

This table defines what a parent sees across every data type. It is the authoritative reference for all API and mobile changes in this spec.

| Data type | Current access | New policy | Rationale |
|---|---|---|---|
| **Session list** (topic, duration, quality, XP) | Full metadata via `GET /dashboard/children/:id/sessions` | **Keep** | Metadata is non-sensitive. Parent should know what topics were covered and for how long |
| **Session transcripts** (turn-by-turn exchanges) | Full raw transcripts via `GET /dashboard/children/:id/sessions/:sid/transcript` | **Remove endpoint** | Raw child input is private conversational space. Summary preserves learning insight without exposing exact wording |
| **Session summary** (`session_summaries.content`) | Available in DB, partially surfaced via `displaySummary` in session list | **Surface explicitly** ŌĆö becomes the primary session detail for parents | Already generated by `session-completed` pipeline. Contains what the child learned without raw transcript |
| **Homework summary** (`homeworkSummary` in session metadata) | Surfaced in `ChildSession.homeworkSummary` | **Keep** | Parent-facing by design ŌĆö the parent needs to know what homework help was given |
| **Progress snapshots** (topics mastered, time, sessions) | Full via `GET /dashboard/children/:id/progress-history` | **Keep** | Aggregate metrics, nothing sensitive |
| **Milestones** | Full via progress routes | **Keep** | Celebrations are parent-friendly by design |
| **Knowledge inventory** (subject/topic breakdown) | Full via `GET /dashboard/children/:id/inventory` | **Keep** | Curriculum-level data, not conversational |
| **Vocabulary** (terms, CEFR, mastery) | Via inventory | **Keep** | Educational content, not private |
| **Mentor memory** (inferences about struggles, preferences, pace) | Full raw profile via `GET /learner-profile/:id` | **Curated categorized view** ŌĆö see Section 3 | Parents need to see what the system "thinks" to correct it, but in human-readable grouped form, not raw inference JSON |
| **Memory "tell" inputs** (parent-sourced) | Visible in learner profile | **Keep** | Parent wrote it, parent should see it |
| **Memory controls** (toggle collection/injection, accommodation mode) | Full control | **Keep** | Parent is GDPR data controller for the child |
| **Streaks / XP** | Not surfaced to parents | **Surface in dashboard** (read-only) | Low-sensitivity motivational data the parent wants to see |
| **Monthly reports** | Full LLM-generated report | **Keep** | Already designed for parent consumption |

## Section 1: Transcript Access Removal

### API changes

**Remove:** `GET /dashboard/children/:profileId/sessions/:sessionId/transcript`

This endpoint currently returns `ChildSessionTranscript` containing `exchanges: TranscriptExchange[]` ŌĆö every `user_message` and `ai_response` event with full content and escalation rung metadata.

**Action:** Remove the route from `apps/api/src/routes/dashboard.ts` and the `getChildSessionTranscript` function from `apps/api/src/services/dashboard.ts`. Remove the `TranscriptExchange` and `ChildSessionTranscript` interfaces if they're only used by this endpoint.

**Replace with:** The session detail view for parents uses the existing `ChildSession` object, which already contains `displaySummary` (from `session_summaries`) and `homeworkSummary`. No new endpoint needed ŌĆö the session list response already carries the summary data.

### Mobile changes

**File:** `apps/mobile/src/app/(app)/child/[profileId]/session/[sessionId].tsx`

This screen currently fetches and renders the transcript. Change it to:
- Show session metadata (topic, duration, quality, XP earned)
- Show `displaySummary` as the session content
- Show `homeworkSummary` if present (homework sessions)
- Remove the transcript exchange list
- If `displaySummary` is null (sessions completed before summary generation was added), show "Session summary not available for older sessions"

### Schema and orphan code cleanup

Transcript-related types and code paths must be removed everywhere, not just at the primary call site. Before the change is merged, grep the repo for every occurrence of the following identifiers and remove matches in schemas, services, route definitions, React Query cache keys, hooks, mobile screens, tests, mock fixtures, and any commented-out JSX:

- `ChildSessionTranscript`
- `TranscriptExchange`
- `getChildSessionTranscript`
- Any route constant or hook keyed to `/dashboard/children/:id/sessions/:sid/transcript`
- Mobile query hooks such as `useChildSessionTranscript` (or the current name)

A single overlooked reference is a runtime crash on the first parent who deep-links to a cached transcript screen. Treat this as part of the implementation, not a follow-up cleanup.

### Client cache invalidation

The mobile app likely has React Query entries cached under transcript query keys. After deploy, the first fetch against the removed endpoint will 404. The session detail screen (Section 1 mobile changes) must handle 404 gracefully and display the summary-only view. No explicit cache purge is required because the screen redesign replaces the data it renders; stale cache entries simply become unused.

## Section 3: Curated Mentor Memory View

### Current state

`GET /learner-profile/:profileId` returns the full `LearnerProfile` object ŌĆö raw inference items, suppressed items, accommodation settings, memory-enabled flags, and parent "tell" inputs. The parent sees everything the system has inferred.

### New state

Create a new endpoint that returns a curated, categorized view of the learner profile for parents.

**New route:** `GET /dashboard/children/:profileId/memory`

**Response shape:**

```typescript
interface CuratedMemoryView {
  categories: MemoryCategory[];
  parentContributions: ParentTellItem[];
  settings: {
    memoryEnabled: boolean;
    collectionEnabled: boolean;
    injectionEnabled: boolean;
    accommodationMode: string | null;
  };
}

interface MemoryCategory {
  label: string;                    // "Struggles with", "Prefers", "Learning pace", "Interests"
  items: CuratedMemoryItem[];
}

interface CuratedMemoryItem {
  // Items have no database-level ID ŌĆö the learning_profiles table stores inferences
  // as JSONB arrays of strings on per-category columns. Deletion is by (category, value).
  category: MemoryCategoryKey;      // which underlying column: 'struggles' | 'interests' | 'strengths' | 'communicationNotes' | 'learningStyle'
  value: string;                    // the inference string itself, e.g. "fractions", "visual analogies"
  statement: string;                // human-readable rendering of value for the parent UI
}

type MemoryCategoryKey =
  | 'struggles'
  | 'interests'
  | 'strengths'
  | 'communicationNotes'
  | 'learningStyle';

interface ParentTellItem {
  id: string;
  content: string;
  createdAt: string;
}
```

### Categorization logic

The `learning_profiles` table (see `packages/database/src/schema/learning-profiles.ts`) already stores inferences on separate JSONB columns ŌĆö one column per category. Categorization is purely a column-to-label mapping, not a classification step. No LLM call, no tag inspection, no `inferenceType` switch.

| Underlying column (`learning_profiles`) | Shape | Parent-facing label |
|---|---|---|
| `struggles` | `string[]` | "Struggles with" |
| `interests` | `string[]` | "Interests" |
| `strengths` | `string[]` | "Strengths" |
| `communicationNotes` | `string[]` | "Learning pace & notes" |
| `learningStyle` | single `object` (serialized to descriptive strings) | "Learning style" |

**Excluded from the curated view:**
- `suppressedInferences` ŌĆö these were already deleted/suppressed by the parent; do not re-surface
- `interestTimestamps`, `recentlyResolvedTopics` ŌĆö internal bookkeeping
- `effectivenessSessionCount`, `version` ŌĆö internal counters

The service reads each column, wraps each entry in `CuratedMemoryItem`, and groups them into `MemoryCategory` objects keyed by the column label. Empty columns yield empty categories and are omitted from the response (the UI shows only non-empty groups).

**`learningStyle` special case:** This column is a single object, not an array. The service serializes its populated fields into one or more descriptive strings (e.g., `{ modality: "visual" }` ŌåÆ `"Prefers visual learning"`). The `value` sent to the delete endpoint is the raw field path (e.g., `"modality"`) so that the existing `deleteMemoryItem` service can null it out. Deletion semantics for `learningStyle` follow whatever `deleteMemoryItem` already supports ŌĆö confirm during implementation.

### Memory deletion

The existing `deleteMemoryItem(db, profileId, category, value, suppress?, subject?)` service function (in `apps/api/src/services/learner-profile.ts`) is the deletion primitive. It takes a `(category, value)` tuple ŌĆö no item IDs involved, because inferences are stored as JSONB array entries rather than individual rows.

The parent taps the [x] button on a curated item; the mobile app sends `{ category, value }` to the existing delete endpoint. The service removes the entry from the relevant column (or, when `suppress = true`, appends it to `suppressedInferences` so the inference engine won't re-infer it).

**Default to `suppress = true` for parent corrections.** When a parent deletes an item, the learner shouldn't re-acquire that inference on the next session. Plain deletion without suppression would let the system re-learn the struggle/preference the parent just said is wrong. This matches the intent of the feature: the parent is stating "the system is wrong about this," not "this was once true and is no longer."

### Secondary escape hatch

If the parent can't find the specific wrong inference in the categorized list, provide a "Something else is wrong" button that opens a simple feedback form. This submits to the existing `POST /learner-profile/:profileId/tell` endpoint with a `source: 'parent_correction'` tag. The system processes it as parent input ŌĆö no automated deletion, just a signal to the learning profile system.

### Mobile changes

**New screen:** `apps/mobile/src/app/(app)/child/[profileId]/mentor-memory.tsx` (already exists ŌĆö modify it)

Replace the current raw profile view with:
- Category groups with short statement items, each with a delete [x] button
- "Your notes" section showing parent "tell" contributions
- Settings toggles (memory on/off, collection, injection, accommodation mode)
- "Something else is wrong?" button at the bottom

## Section 4: RLS Policy for `family_links`

### Current state

`family_links` has RLS enabled (migration `0027`) but zero policies. The application layer enforces access via `assertParentAccess` (checks `findFirst` where `parent_profile_id = parentId AND child_profile_id = childId`).

### Policy

Add to the RLS Phase 2 migration (`0026_rls_policies.sql` as defined in `2026-04-15-S06-rls-phase-2-4-enforcement.md`):

```sql
-- Parent can read their own family links (both directions needed for consent UI)
CREATE POLICY family_parent_access ON family_links
  FOR SELECT TO app_user
  USING (
    parent_profile_id = current_setting('app.current_profile_id', true)::uuid
  );

-- Child can read links where they are the child (needed for consent UI to show who granted consent)
CREATE POLICY family_child_access ON family_links
  FOR SELECT TO app_user
  USING (
    child_profile_id = current_setting('app.current_profile_id', true)::uuid
  );

-- No INSERT/UPDATE/DELETE via app_user ŌĆö family link creation/removal
-- stays in the consent service, which runs as owner
```

**Design decision:** Separate SELECT policies for parent and child (rather than the current plan's bidirectional `OR` policy) because:
- The parent needs to see all their children
- The child needs to see who their parent is (consent UI)
- Neither needs to INSERT/UPDATE/DELETE ŌĆö that's handled by the consent service which will use `ownerDb`

**Implicit write denial:** No `FOR INSERT`, `FOR UPDATE`, or `FOR DELETE` policies are declared for the `app_user` role on `family_links`. In Postgres, a role with RLS enabled and no matching policy for a command is implicitly denied. That means any attempt by `app_user` to write to `family_links` (e.g., a future service bug that uses the scoped DB instead of `ownerDb`) is rejected at the database layer with `permission denied for table family_links`. The consent service must continue to run as the owner role ŌĆö typically via `ownerDb` ŌĆö to perform link creation, acceptance, and revocation.

### Update to Phase 2-4 plan

Replace the existing `family_access` policy in `2026-04-15-S06-rls-phase-2-4-enforcement.md` with the two policies above. Remove the `WITH CHECK` clause ŌĆö writes go through `ownerDb`.

## Section 5: Parent-Access Subquery Policies (RLS Phase 2 Update)

### Problem

The Phase 2 plan writes `profile_isolation` policies on all child-data tables:

```sql
CREATE POLICY profile_isolation ON learning_sessions
  FOR ALL TO app_user
  USING (profile_id = current_setting('app.current_profile_id', true)::uuid)
```

Once Phase 3 enforces RLS, the parent dashboard breaks. The dashboard service queries `learning_sessions`, `session_summaries`, `progress_snapshots`, etc. using the child's `profile_id` ŌĆö but the parent's `app.current_profile_id` is set to the *parent's* profile, not the child's.

### Solution

Add parent-read policies to tables the parent dashboard reads. The parent's SELECT includes a subquery against `family_links`.

**Tables that need parent-read policies:**

| Table | Parent reads via | Policy needed |
|---|---|---|
| `learning_sessions` | `getChildSessions` | SELECT where child is linked |
| `session_summaries` | `getChildSessions` (join) | SELECT where child is linked |
| `progress_snapshots` | `getChildProgressHistory` | SELECT where child is linked |
| `milestones` | progress routes | SELECT where child is linked |
| `streaks` | `getChildDetail` (new) | SELECT where child is linked |
| `xp_ledger` | `getChildDetail` (new) | SELECT where child is linked |
| `vocabulary` | `getChildInventory` | SELECT where child is linked |
| `vocabulary_retention_cards` | `getChildInventory` | SELECT where child is linked |
| `subjects` | `getChildSubjectTopics` | SELECT where child is linked |
| `retention_cards` | `getChildInventory` | SELECT where child is linked |
| `assessments` | `getChildInventory` | SELECT where child is linked |

**Tables that do NOT get parent-read policies:**

| Table | Reason |
|---|---|
| `session_events` | **Privacy boundary** ŌĆö parents do not see raw transcripts |
| `session_embeddings` | Internal vector data, not parent-facing |
| `parking_lot_items` | Child's private queue |
| `onboarding_drafts` | Transient setup data |
| `coaching_card_cache` | Child-facing coaching, not parent-facing |
| `notification_preferences` | Child's notification settings |
| `notification_log` | Child's notification history |
| `learning_modes` | Internal state |
| `needs_deepening_topics` | Internal retention logic |
| `teaching_preferences` | Internal pedagogy settings |
| `curriculum_adaptations` | Internal curriculum state |

**Policy template for parent-read tables:**

```sql
-- Add alongside existing profile_isolation policy (do not replace it)
CREATE POLICY parent_read_via_family ON learning_sessions
  FOR SELECT TO app_user
  USING (
    profile_id IN (
      SELECT child_profile_id FROM family_links
      WHERE parent_profile_id = current_setting('app.current_profile_id', true)::uuid
    )
  );
```

This policy is additive ŌĆö Postgres OR's multiple policies together for the same role and command. The child's own `profile_isolation` policy still works. The parent's `parent_read_via_family` policy allows SELECT on rows belonging to their linked children.

**Performance note:** The `family_links` subquery runs on every SELECT against these tables when the user is a parent. The existing index `family_links_child_profile_id_idx` covers lookups by child. Add an index on `parent_profile_id` if one doesn't exist:

```sql
CREATE INDEX IF NOT EXISTS family_links_parent_profile_id_idx
  ON family_links (parent_profile_id);
```

### Pre-implementation verification (required before writing migrations)

The table list above is derived from a brainstorm-phase code walk, not an exhaustive audit. A missed table under Phase 3 RLS enforcement means the parent dashboard returns empty or 500s for that data surface, silently, in production. An included-but-unread table means a policy with no consumer (low-cost but noise).

**Do before writing the Phase 2 migration edit:**

1. Enumerate every table read by `apps/api/src/services/dashboard.ts` and its transitive service imports (e.g., whatever `getChildInventory` calls, whatever `getChildProgressHistory` reaches into).
2. Cross-reference each read table against the "needs parent-read policy" list in Section 5.
3. For every table in the "does NOT get parent-read policies" list, confirm no dashboard read path touches it. A grep for the table name across `apps/api/src/services/` is sufficient.
4. Record the verification result inline in the implementation plan ŌĆö list each table and the service function(s) that read it, or "no parent-visible reader" for excluded tables.

Do not write the Phase 2 migration update until this verification is complete and recorded.

### Deliverable

Update `2026-04-15-S06-rls-phase-2-4-enforcement.md` Phase 2 section to include:

4. The `parent_profile_id` index

## Section 6: Consent & Learner Profile Route Adjustments

### Learner profile parent routes

The parent-scoped `GET /learner-profile/:profileId` endpoint currently returns the full raw profile. Two options:

**Option A (recommended):** Keep the existing endpoint as-is (it serves the self-learner use case too) and route parent mobile screens to the new `GET /dashboard/children/:profileId/memory` endpoint instead. The raw endpoint still exists but the parent UI never calls it.

**Option B:** Add a middleware check that returns the curated view when the caller is a parent. More complex, mixes concerns.

Go with Option A. The raw endpoint is still available for internal/debug use but the parent mobile screens use the curated endpoint.

### No changes to write endpoints

All parent write endpoints (`DELETE /learner-profile/:profileId/item`, `PATCH .../memory-enabled`, `POST .../tell`, etc.) stay unchanged. The parent can still delete items, toggle settings, and submit observations. The curated view is read-only presentation ŌĆö writes go through the existing endpoints.

## Failure Modes

| State | Trigger | User sees | Recovery |
|---|---|---|---|
| Parent navigates to old transcript URL (deep link / bookmark) | Endpoint removed | 404 response | Mobile handles 404 ŌåÆ shows "Session details" with summary instead of transcript. If summary is null: "Summary not available for older sessions" |
| `displaySummary` is null for old sessions | Sessions completed before summary generation existed | "Summary not available for older sessions" | No action needed ŌĆö historical data gap, not a bug |
| Curated memory view has zero items | Child has no sessions yet, or memory is disabled | "No learning observations yet. As [Name] uses the app, the mentor will learn about their preferences and pace." | No action ŌĆö informative empty state |
| Parent deletes a memory item that was already deleted | Race condition (two parents, or stale UI) | Delete endpoint returns success (idempotent) | No error shown ŌĆö item disappears from list on refresh |
| `family_links` RLS policy blocks legitimate parent read | Misconfigured `app.current_profile_id` or missing family link row | Empty children list in dashboard | Dashboard shows "No linked children found" with a "Refresh" button. If persistent: "Contact support" |
| Parent views dashboard with child profile deleted | `CASCADE` delete on `family_links` removes the link | Child disappears from list | Dashboard re-fetches, child is gone. No error ŌĆö the link correctly no longer exists |
| Parent views dashboard offline | No network | Cached data shown (React Query cache) + "You're offline" banner | Banner with "Retry when connected" |
| `parent_read_via_family` subquery returns wrong children | Bug in family link creation | Parent sees wrong child's data | Break test required: integration test that creates two families, verifies isolation |
| "Something else is wrong" submission fails | Network error on `POST .../tell` | Toast: "Couldn't submit your feedback. Tap to retry." | Retry button in toast |

- **Full RLS policy sweep** ŌĆö this spec adds parent-access policies only. The comprehensive RLS audit for all roles stays in the Phase 2-4 plan.

## Test Plan

### Break tests (security)

1. **Transcript endpoint gone:** Request `GET /dashboard/children/:id/sessions/:sid/transcript` ŌåÆ 404
2. **Curated memory isolation:** Parent A cannot see Parent B's child's memory via `GET /dashboard/children/:id/memory` ŌåÆ 403
3. **Family link RLS:** With `app.current_profile_id` set to an unrelated profile, `SELECT * FROM family_links` returns zero rows
4. **Parent-read RLS isolation:** With parent's profile_id set, `SELECT * FROM learning_sessions` returns only linked children's sessions, not all sessions
5. **session_events excluded:** With parent's profile_id set, `SELECT * FROM session_events` returns zero rows (even for linked children)

### Functional tests

1. Session list returns `displaySummary` for each session
2. Curated memory groups inferences by category correctly
3. Memory item deletion via [x] button works end-to-end
4. Streaks and XP appear in dashboard child detail
5. "Something else is wrong" submits to tell endpoint with `parent_correction` tag
6. Sessions with null `displaySummary` show fallback copy

### Mobile tests

1. Session detail screen renders summary, not transcript
2. Mentor memory screen shows categorized items with delete buttons
3. Dashboard child detail shows streak/XP stats
4. Empty states render correctly for all screens

# Learner Experience Features — Session Recap, Bookmarks, Encouragement

**Date:** 2026-04-22
**Status:** Draft — awaiting review
**Scope:** Three features addressing learner engagement gaps

## Problem Statement

After using the app as a learner, three experience gaps are clear:

1. **No learning visibility.** The session summary shows mechanical stats (time, exchanges, escalation rung) but never says *what you learned* or *what's next*. The session just ends.
2. **No way to save moments.** When the AI explains something beautifully, the learner can't bookmark or save it. Messages scroll by and are only retrievable by navigating library → book → past session.
3. **No encouragement.** The prompt actively bans praise phrases and instructs the LLM to "say nothing about correctness and just continue teaching." For 11–17 year olds, getting silence after a correct answer is deflating.

---

## Feature 3: Encouragement — Specific, Earned Recognition

**Type:** Prompt-only change (no DB, no API, no UI changes)

### Root Cause

The Prohibitions block in `exchange-prompts.ts:584–594` correctly bans performative phrases ("Great job!", "Amazing!") but overcorrects by leaving no middle ground between "That's correct" and silence. The current instruction is:

> "Acknowledge progress factually and vary it: 'That's correct', 'Yes', 'You've got it', or just move on. Sometimes say nothing about correctness and just continue teaching."

This produces a tutor that *never* tells you what you did well. The fix is specific, earned recognition — not generic praise.

### Design

Replace the blanket "factual acknowledgment only" instruction with a two-tier encouragement framework matching the product's 11+ user base.

**Tier 1 — Ages 11–14 (early teens):**

```
When the learner makes a correct connection or shows understanding, name what
they got right: "You just linked respiration back to the energy cycle — that's
the key insight." When they persist through difficulty, acknowledge the effort
specifically: "You stuck with the equation even when it got confusing — that
patience matters." Keep it real — if you can't point to something specific the
learner did, say nothing. Never generic.
```

**Tier 2 — Ages 15–17 (teens):**

```
Acknowledge strong reasoning or unexpected connections briefly: "Good catch",
"That's a sharp connection", "Exactly right, and here's why that matters..."
Deliver it and move forward — don't linger on praise. Never patronize.
```

**Session-closing encouragement:**

The in-exchange encouragement above addresses the mid-session gap, but the session *ending* is the moment a learner most wants recognition. The session summary screen should open with a brief closing line generated alongside the recap (Feature 1). This line mirrors what the learner did — not a grade, not generic praise:

- **Tier 1 (11–14):** "You worked through 3 problems on quadratic equations and stuck with it when factoring got tricky."
- **Tier 2 (15–17):** "Solid session — you connected trade policy back to supply chain mechanics."

This line is generated by the same Inngest step as the learner recap (Feature 1, `generate-learner-recap`). It's stored as a new `closingLine` column on `session_summaries` and displayed at the top of the summary screen, above the "What happened" stats card. Subject to the same `exchangeCount >= 3` threshold — short sessions don't get a closing line.

**Still banned (all ages):** "Great job!", "Amazing!", "I'm so proud of you!", "Fantastic!", "Awesome!", "Nice work!", "Excellent!", "Let's dive in!" — these remain banned because they're non-specific and performative.

**Still kept (all ages):** "Not yet" framing for incorrect answers. No "wrong"/"incorrect"/"mistake". Acknowledge partial correctness before guiding further.

## Feature 2: Bookmarks — Save This Moment

**Type:** New DB table + API routes + mobile UI

### Design

Lightweight toggle on individual AI messages. No highlighting within messages, no tags, no categories, no sharing. Bookmark/unbookmark a whole AI response.

#### Database

New table `bookmarks`:

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid (UUIDv7) | PK |
| `profileId` | uuid | FK → profiles, indexed |
| `sessionId` | uuid | FK → learning_sessions |
| `eventId` | uuid | FK → session_events (the AI response event) |
| `subjectId` | uuid | FK → subjects (denormalized for query efficiency) |
| `topicId` | uuid (nullable) | FK → curriculum_topics (nullable for freeform sessions) |
| `content` | text | Snapshot of AI message text at bookmark time |
| `createdAt` | timestamp | Default now() |

**Unique constraint:** `(profileId, eventId)` — can't bookmark the same message twice.

**Content snapshotting:** The `content` column stores a **snapshot** of the AI message text at bookmark time, copied from `session_events.content`. This is a deliberate denormalization — bookmarks must survive session cleanup/TTL. If the session is deleted later, the bookmark still displays the original explanation. Without this, bookmarks that join live `session_events` data would silently break when sessions age out, violating the core promise of "save this moment." The `eventId` FK is kept for provenance but the joined content is not relied upon for display.

**Pattern note:** The existing feedback system records `user_feedback` events in `session_events` (via `handleMessageFeedback` in `use-session-actions.ts:501`). Bookmarks use a separate table because they outlive session context — a learner revisits bookmarks across sessions and subjects, which requires its own query path with efficient joins. Session events are session-scoped and don't support cross-session listing efficiently.

#### API Routes

New route group in `apps/api/src/routes/bookmarks.ts`:

| Method | Path | Body/Query | Response |
|--------|------|------------|----------|
| POST | `/bookmarks` | `{ eventId }` | `{ bookmark }` (201) |
| DELETE | `/bookmarks/:id` | — | 204 |
| GET | `/bookmarks` | `?cursor=&limit=20&subjectId=` | `{ bookmarks[], nextCursor }` |

- POST resolves `sessionId`, `subjectId`, `topicId` from the `session_events` row server-side and snapshots `content` from `session_events.content` (learner only sends `eventId`)
- GET returns bookmarks directly — `content` is denormalized on the bookmark row, no session_events join needed
- All endpoints scoped by `profileId` from auth context (reads via `createScopedRepository(profileId)`, writes include explicit `profileId` protection)
- Pagination via cursor (UUIDv7 is naturally time-ordered)
- **Parent visibility:** Parent accounts cannot access child bookmarks in v1. Bookmarks are personal to the learner profile. If parent visibility is added later (see `docs/specs/2026-04-18-parent-progress-highlights.md`), bookmarks would be a natural extension — the scoped repository pattern supports it

#### Schemas Package

New `bookmarkSchema` in `packages/schemas/src/bookmarks.ts`:

```typescript
export const bookmarkSchema = z.object({
  id: z.string().uuid(),
  eventId: z.string().uuid(),
  sessionId: z.string().uuid(),
  subjectId: z.string().uuid(),
  topicId: z.string().uuid().nullable(),
  subjectName: z.string(), // joined
  topicTitle: z.string().nullable(), // joined
  content: z.string(), // snapshotted AI message text (survives session deletion)
  createdAt: z.string().datetime(),
});

export const createBookmarkSchema = z.object({
  eventId: z.string().uuid(),
});
```

#### Mobile UI

**Bookmark button placement:**

The existing `SessionMessageActions` renders feedback pills (Helpful / Not helpful / Incorrect) in a `flex-row flex-wrap gap-2` layout. On a 5.8" Galaxy S10e, adding a 4th text pill is crowded.

**Solution:** Render the bookmark as an **icon-only button** (Ionicons `bookmark-outline` / `bookmark`) positioned to the **right of the feedback row**, visually separated. This avoids crowding the text pills while keeping the bookmark discoverable. The icon toggles between outline (unsaved) and filled (saved) states.

```
[Helpful] [Not helpful] [That's incorrect]    [🔖]
```

- Haptic feedback on toggle (`Haptics.impactAsync(ImpactFeedbackStyle.Light)`)
- Optimistic UI: toggle immediately, revert on API failure + show toast "Couldn't save bookmark. Check your connection."
- Track bookmark state per-message in a local `Record<string, string | null>` (eventId → bookmarkId or null)
- On session load, fetch bookmarks for the current session to pre-populate filled states

**Past-session bookmarking:** The bookmark icon also renders when reviewing past sessions from the library (session detail/replay view), not only during live sessions. The data model supports this — `eventId` + `profileId` is all that's needed. Learners frequently realize an explanation was valuable *after* the session ends, and the only current path (library → book → past session) should allow saving, not just reading.

**Discoverability:**

An 11-year-old in their first session won't scan for a small icon among feedback pills. Without prompting, the feature risks near-zero adoption.

- **First-session nudge:** After the 3rd AI response in a learner's *first ever session* (check: profile has zero prior sessions), show a one-time tooltip pointing at the bookmark icon: *"Tap to save explanations you want to revisit."* Dismissable, stored in SecureStore (`bookmark-nudge-shown`), never shown again.
- **Post-session prompt:** If the session had `exchangeCount >= 5` and the learner saved zero bookmarks, the recap screen (Feature 1) shows a subtle line below the "What you explored" card: *"Some great explanations in this session — you can bookmark them next time."* Only shown for the first 3 sessions with zero bookmarks, then stops (not nagging).

**Saved screen:** New route `/(app)/progress/saved.tsx`

- Natural sibling to milestones (`/(app)/progress/milestones.tsx`) and vocabulary (`/(app)/progress/vocabulary.tsx`)
- Entry point: new row in the progress index screen, between milestones and "Keep learning"
- FlatList with cursor pagination
- Each row: AI message content (5-line truncation — 3 lines cuts off step-by-step explanations at the setup, hiding the punchline), subject name, topic title, relative date
- Tap → expand to show full exchange (user question + AI response)
- Swipe-to-delete with confirmation

**Empty state:** "Nothing saved yet. Tap the bookmark icon on any response during a session to save it here."

## Feature 1: Session Recap — What You Covered & What's Next

**Type:** Inngest pipeline extension + new DB columns + mobile UI

### Design

Add a **learner-facing recap** to the `session.completed` Inngest pipeline. The parent-facing pipeline already generates `highlight`, `narrative`, `conversationPrompt`, and `engagementSignal` — we add a parallel learner-facing step.

#### What the Learner Sees

**Closing line** (top of summary screen, above "What happened" stats card):

A single sentence mirroring what the learner did — not a grade, not generic praise. Generated by the same Inngest step as the recap bullets. Uses the same age-tier voice as the encouragement feature (Feature 3):

- **Tier 1 (11–14):** "You worked through 3 problems on quadratic equations and stuck with it when factoring got tricky."
- **Tier 2 (15–17):** "Solid session — you connected trade policy back to supply chain mechanics."

Stored as `closingLine` on `session_summaries`. Subject to the same `exchangeCount >= 3` threshold.

**Three new cards on the session summary screen** (`/session-summary/[sessionId].tsx`):

**1. "What you explored" card** (between "What happened" and milestones):
- 2–4 bullet points extracted from the transcript: concepts covered, questions answered, connections made
- Written in second person, age-appropriate voice
- Example: "You figured out how the Calvin cycle uses CO₂ to build glucose"
- NOT a transcript summary — these are *learning takeaways*

**2. "Up next" card** (below milestones, above "Your Words"):
- For **curriculum-attached sessions** (has a `topicId`): shows the next unstarted topic in the same book — title + one-sentence connection to what was just covered. "Continue learning" CTA → navigates to start a session on that topic.
- For **freeform sessions**: shows a **"You might also like..."** variant with 1–2 related curriculum topics matched by comparing the session's extracted concepts (from the recap bullets) against topic titles/descriptions via a lightweight DB text search. Low-confidence match (no results or relevance score below threshold) → card not shown. This avoids fabricating a suggestion while still giving curious freeform learners a forward path into real curriculum content.

**3. Bookmark prompt** (conditional, below "Up next"):
- Only shown when: `exchangeCount >= 5`, learner saved zero bookmarks in this session, and this is one of the learner's first 3 qualifying sessions (prevent nagging)
- Text: *"Some great explanations in this session — you can bookmark them next time."*
- Helps drive bookmark discoverability (see Feature 2)

#### Minimum Threshold

Both cards require `exchangeCount >= 3`. A 1–2 exchange session doesn't have enough material for meaningful takeaways. Below the threshold, neither card appears — the summary screen shows the existing "What happened" stats card only (status quo).

#### Database

New columns on `session_summaries` (following the existing pattern of plain text columns):

| Column | Type | Notes |
|--------|------|-------|
| `closingLine` | text (nullable) | Single sentence mirroring what the learner did (age-tiered voice) |
| `learnerRecap` | text (nullable) | Markdown bullet list: `- You explored...\n- You connected...` |
| `nextTopicId` | uuid (nullable) | FK → curriculum_topics |
| `nextTopicReason` | text (nullable) | One sentence: "This builds on the photosynthesis process you just studied" |

**Why text, not jsonb:** Every existing insight column on `session_summaries` is plain text (`highlight`, `narrative`, `conversationPrompt`, `engagementSignal`). Introducing jsonb would break the established pattern. The mobile client parses `learnerRecap` by splitting on `\n- ` — simple and consistent.

#### Inngest Pipeline

New step in `session-completed.ts`, added after the existing `generateSessionInsights` step:

```
Step: "generate-learner-recap"
Guard: exchangeCount >= 3 (topicId may be null for freeform — recap still generated, but nextTopic fields stay null)
Input: session transcript (already loaded by prior step)
LLM call: Haiku-tier via services/llm/router.ts, fast (<3s)
Output: { closingLine: string, learnerRecap: string, nextTopicId: string | null, nextTopicReason: string | null }
```

**Structured output:** The LLM call uses a dedicated Zod schema (`learnerRecapResponseSchema` in `@eduagent/schemas`) with `zodResponseFormat` for structured JSON output. This avoids parsing markdown from free-text and aligns with the architecture rule against embedding JSON blobs in free-text replies. The response envelope is *not* used here — this is an informational extraction, not a state-machine signal.

**Recap generation prompt** (Haiku-tier):

```
You are reviewing a tutoring session transcript. Produce a closing line and 2–4 learning takeaways.

Closing line rules:
- One sentence mirroring what the learner specifically did in this session
- Name the subject matter and what made the session noteworthy (persistence, a connection, a breakthrough)
- Not a grade, not generic praise — a mirror
- Adapt tone to learner age: [ageVoiceTier]
- Max 150 characters

Takeaway rules:
- Write in second person ("You explored...", "You connected...", "You figured out...")
- Each takeaway must name a specific concept or skill, not generic ("learned stuff")
- Format as a markdown bullet list (- prefix)
- Adapt language complexity to the learner's age: [ageVoiceTier]
- Max 200 characters per bullet
- No praise, no filler — just what was covered
```

**Next topic resolution** (no LLM — pure DB query):

```sql
SELECT ct.id, ct.title
FROM curriculum_topics ct
WHERE ct.book_id = :currentBookId
  AND ct.sort_order > :currentSortOrder
  AND ct.skipped = false
  AND ct.id NOT IN (SELECT topic_id FROM retention_cards WHERE profile_id = :profileId)
ORDER BY ct.sort_order ASC
LIMIT 1
```

If no next topic in the current book → check next book in subject. If curriculum complete → `nextTopicId` stays null (card not shown).

**Query correctness note:** The `NOT IN (SELECT topic_id FROM retention_cards ...)` subquery filters topics that have generated retention cards. Verify during implementation that this correctly identifies "already studied" topics — if a topic was started (has a learning session) but hasn't yet generated retention cards, it would incorrectly appear as "next." May need to also check `learning_sessions` for topic attachment as a fallback.

**Freeform topic matching** (no LLM — text search):

For freeform sessions (`topicId IS NULL`), extract concept keywords from the recap bullets and match against `curriculum_topics.title` and `curriculum_topics.description` using a basic `ILIKE` or `ts_vector` search:

```sql
SELECT ct.id, ct.title, ct.description
FROM curriculum_topics ct
JOIN books b ON b.id = ct.book_id
JOIN subjects s ON s.id = b.subject_id
WHERE s.profile_id = :profileId
  AND (ct.title ILIKE ANY(:keywords) OR ct.description ILIKE ANY(:keywords))
LIMIT 2
```

If zero results or the match quality is too low (e.g., only single-word generic matches like "science") → card not shown. This keeps freeform suggestions grounded in real curriculum content without relying on LLM hallucination.

**Next topic reason** generation (Haiku-tier, appended to recap call):

```
Also suggest why the next topic "[nextTopicTitle]" connects to what was just covered.
One sentence, max 120 characters. Example: "This builds on the energy cycle you just explored."
```

#### Mobile UI — Real-Time Loading Pattern

**Challenge:** The session summary screen navigates via `router.replace` with data passed as URL params. The `useSessionSummary` hook exists but is used for re-entry to past sessions. For the initial post-session flow, the Inngest job hasn't completed yet.

**Solution:** Poll with `refetchInterval` until the recap fields are populated:

```typescript
const summary = useSessionSummary(sessionId, {
  // Poll every 2s until learnerRecap arrives (Inngest job takes 3-8s)
  refetchInterval: (data) =>
    !isAlreadyPersisted && !data?.learnerRecap ? 2000 : false,
});
```

- On first render: "What you explored" and "Up next" cards show shimmer skeletons
- When `summary.data?.learnerRecap` becomes non-null: replace skeletons with content
- After 15 seconds with no data: stop polling, hide the skeleton cards (graceful degradation — the learner still sees the existing summary content)
- For re-entry to past sessions: data loads immediately from the persisted row (no polling needed)

## Learning Loop Vision

These three features are designed to be independently shippable, but the real learner value comes from them **connecting into a learning loop**:

```
Encouragement during session
  → Recap + closing line at session end
    → Bookmarks that persist across sessions
      → Recap feeds context into the NEXT session
        → Encouragement references past progress
          → (loop)
```

This spec is **v1 — the foundation**. The features are independent today but share data that enables the loop:

| v1 (this spec) | v2 (future — enabled by v1 data) |
|----------------|----------------------------------|
| Encouragement is per-exchange, context-free | Encouragement references past sessions: "Last time you struggled with factoring — you've clearly practiced since then" |
| Recap bullets are generated and displayed | Recap bullets are fed into the next session's exchange prompt as prior context: "In your last session, you covered: [bullets]" |
| Bookmarks are a personal save feature | Bookmarked explanations surface as review prompts: "You saved an explanation about the Calvin cycle 2 weeks ago — want a quick refresher?" |
| Closing line mirrors the current session | Closing line references the arc across sessions: "Three sessions in, and you've gone from basic cell structure to full photosynthesis" |

**v1 implementation decisions shaped by v2:**

1. **`learnerRecap` is stored as text** — but the structured Zod schema (`learnerRecapResponseSchema`) means it can be parsed programmatically for injection into exchange prompts. No format change needed for v2.
2. **Bookmark `content` is snapshotted** — survives session cleanup, so bookmarked explanations remain available as review material even after session TTL.
3. **`closingLine` is a separate column** — can be read independently by the exchange prompt builder without loading the full summary.
4. **Encouragement is tiered by age** — the tier framework extends naturally to include "references past progress" as a third dimension alongside "specific" and "brief."

Naming this loop now ensures v1 implementation choices (data shapes, storage decisions, column granularity) don't create expensive retrofitting later.

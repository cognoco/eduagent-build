# Quiz Activities Design

> Three playful learning activities — Capitals, Vocabulary, Guess Who — sharing a common quiz engine. Discovery-first games that silently reinforce mastery.

## 1. Philosophy

These are **discovery-first games**, not tests. Most questions introduce new content the learner hasn't studied. A configurable ratio (~25%) of questions are silent mastery checks pulled from the learner's library — same domain, same UI, the kid never knows the difference.

Wrong answers on discovery questions are silently saved and surfaced later as coaching cards on the home screen. **Never on the results screen.** The results screen has exactly two actions: Play Again or Done.

## 2. Architecture

Standalone system like Dictation — own tables, own routes, no chat sessions. One LLM call generates an entire round upfront as structured JSON. The client renders the quiz locally with no per-question API calls.

```
Practice Menu → Pick activity → LLM generates round → Client renders quiz → Results → SM-2 updates + missed items saved
```

**Pre-fetching is launch scope:** While the learner plays question 3 (midpoint), the client fires a prefetch call for the next round. By the time they finish, the next round loads instantly.

## 3. Activity Types

### 3.1 Capitals

- Multiple choice (4 options)
- Content: "What is the capital of [country]?"
- Themed rounds: "Central European Capitals", "Southeast Asian Capitals", etc.
- **Source of truth:** `capitals_reference` static table (~250 rows), not the LLM
- LLM generates: theme selection, fun facts, distractors
- Validation: every LLM-generated "correct answer" cross-checked against reference table

### 3.2 Vocabulary

- Multiple choice (4 options)
- Content: "Translate: _der Hund_" → pick the correct translation
- Themed rounds: "German Animals", "French Food", etc.
- Scoped by learner's target language and CEFR level
- **Only visible** when the learner has at least one language subject
- Mastery questions pulled directly from the learner's vocabulary bank (never LLM-generated)
- Discovery questions generated by LLM within CEFR level + 1

### 3.3 Guess Who (Clue-by-Clue)

- Free text input with progressive clue reveal
- 5 clues per person, ordered hardest → easiest
- Clue 5 must be a near-giveaway
- No clue may contain the person's name
- **After clue 3:** multiple choice fallback (4 names) appears below the text input. Both input methods active.
- Person selection prefers people from the learner's studied topics; falls back to an age-appropriate general pool
- Fewer questions per round (4 instead of 6-8) because each question is longer

**Scoring per Guess Who question:**

| Guessed after | Points |
|---|---|
| Clue 1 | 5 |
| Clue 2 | 4 |
| Clue 3 | 3 |
| Clue 4 (MC available) | 2 |
| Clue 5 (MC available) | 1 |
| MC tap (clue 4-5) | 1 |
| Skip / wrong after all clues | 0 |

**Fuzzy answer matching** — scaled by name length:

```typescript
const maxDistance = Math.max(1, Math.floor(name.length / 4));
```

| Name length | Max Levenshtein distance | Example |
|---|---|---|
| 4 ("Bach") | 1 | "Bash" OK (distance 1), "Bahc" rejected (distance 2) |
| 6 ("Newton") | 1 | "Newten" OK |
| 8 ("Einstein") | 2 | "Einstien" OK |
| 11 ("Tchaikovsky") | 2 | "Tchaikovski" OK |

## 4. Two-Tier Question Model

### Discovery questions (majority of round)

- Content the learner may not have studied yet
- Wrong answer → show correct answer + fun fact → move on. No penalty.
- Wrong answers silently saved to `quiz_missed_items` for later coaching card surfacing
- No SM-2 update

### Mastery questions (configurable ratio, default ~25%)

- Pulled from the learner's library, **same domain as current activity**
- Playing Capitals → mastery question is also a capital. Never cross-domain mixing.
- Same UI, same energy — the learner doesn't know it's a mastery check
- Correct/incorrect feeds SM-2 retention algorithm
- If no same-domain library content exists, the slot becomes a discovery question

### Library ratio configuration

```typescript
export const QUIZ_CONFIG = {
  defaults: {
    roundSize: 6,
    libraryRatio: 0.25,
    libraryRatioMinItems: 3,          // below → skip mastery questions entirely
    libraryRatioScaleUpThreshold: 20, // above this many due items → bump to 0.35
    timerEnabled: true,
    timerBonusThresholdMs: 5000,
    recentlySeenBufferSize: 30,
  },
  perActivity: {
    capitals: { roundSize: 8 },
    vocabulary: { roundSize: 6 },
    guess_who: { roundSize: 4 },
  },
} as const;
```

The ratio scales with library size:
- `< 3` library items in domain → mastery slots = 0, pure discovery
- `3-20` due items → default ratio (25%)
- `> 20` due items → increased ratio (35%)

## 5. Data Model

### `quiz_activity_type` enum

```sql
'capitals' | 'vocabulary' | 'guess_who'
```

### `quiz_rounds` table

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `profile_id` | uuid FK → profiles | Scoped read/write |
| `activity_type` | quiz_activity_type enum | |
| `theme` | text | "Central European Capitals" |
| `questions` | jsonb | Full round content (see note below) |
| `results` | jsonb | Per-question: correct/incorrect, answer given, time_ms |
| `score` | integer | Correct count |
| `total` | integer | Question count |
| `xp_earned` | integer | |
| `library_question_indices` | integer[] | Which positions were mastery checks |
| `status` | text | `active`, `completed`, `abandoned` |
| `created_at` | timestamptz | |
| `completed_at` | timestamptz | nullable |

> **Known denormalization:** `questions` as JSONB stores the full round content in one blob. This means you cannot query individual questions across rounds (e.g., "which capitals has this learner seen?") without parsing JSON server-side. The `recently_seen` buffer works around this for deduplication. If per-question analytics across rounds become necessary, extract to a `quiz_questions` child table. Not a blocker for Phase 1.

**Abandoned rounds cannot be resumed.** A half-finished round is a spoiled deck — the learner has already seen correct answers via feedback on earlier questions. Resuming would let them answer from memory of the feedback, not from knowledge. This is intentional, not a missing feature.

## 6. Content Generation & Validation

### Content Resolver (shared logic, runs before LLM call)

```
1. Check library size for this activity type + domain
2. Calculate mastery slot count = floor(library_ratio x round_size)
   - library items < libraryRatioMinItems → mastery slots = 0
   - due items > scaleUpThreshold → ratio = 0.35
3. Pull due mastery items (same-domain, ordered by SM-2 next_review_at)
4. Pull recently_seen buffer (last ~30 answer strings from recent rounds)
5. Build LLM prompt: activity type, age bracket, theme preferences,
   discovery count needed, items to exclude
6. LLM generates discovery questions as structured JSON
7. Validate LLM response (per-activity rules below)
8. Inject mastery questions at random positions (not always the same slot)
9. Return complete round to client
```

**Design principle:** Mastery questions are never LLM-generated. They are pulled directly from reference tables (capitals) or the learner's vocabulary bank and formatted into the same question schema. This ensures mastery checks are always factually correct by construction. The LLM only generates discovery content, where wrong answers have zero stakes.

### Capitals validation

- Every "correct answer" cross-checked against `capitals_reference`
- LLM answer mismatch → substitute reference answer silently
- No distractor may be the correct capital for its own country
- No two distractors from the same region
- Accepted aliases checked on answer evaluation ("Czech Republic" = "Czechia")

### Vocabulary validation

- Schema requires `term`, `translation`, `acceptedTranslations[]`
- Prompt instruction: "distractors must not be synonyms or partial translations of the correct answer"
- CEFR level of generated content must not exceed learner's current level + 1

### Guess Who validation

- Schema requires `canonicalName`, `acceptedAliases[]`, exactly 5 clues ordered hard → easy
- No clue may contain the person's name (prompt instruction + post-validation grep)
- Clue 5 must be near-giveaway level

### Question schemas

```typescript
type CapitalsQuestion = {
  type: 'capitals';
  country: string;
  correctAnswer: string;
  acceptedAliases: string[];
  distractors: string[];      // exactly 3
  funFact: string;
  isLibraryItem: boolean;     // hidden from client rendering
  topicId?: string;           // for SM-2 update if library item
};

type VocabularyQuestion = {
  type: 'vocabulary';
  term: string;               // "der Hund"
  correctAnswer: string;      // "dog"
  acceptedAnswers: string[];  // ["dog", "the dog"]
  distractors: string[];      // exactly 3
  funFact: string;
  cefrLevel: string;
  isLibraryItem: boolean;
  vocabularyId?: string;      // for SM-2 update if library item
};

type GuessWhoQuestion = {
  type: 'guess_who';
  canonicalName: string;         // "Isaac Newton"
  acceptedAliases: string[];     // ["Newton", "Sir Isaac Newton"]
  clues: string[];               // exactly 5, hardest first
  mcFallbackOptions: string[];   // 4 names, shown after clue 3
  funFact: string;
  isLibraryItem: boolean;
  topicId?: string;
};
```

## 7. LLM Model Routing

**Default: Gemini Flash** — fast, cheap, sufficient for factual generation with structured output.

**Per-activity override:** If structured output compliance drops below 90% on Guess Who clue generation during development, switch Guess Who to a more capable model (e.g., Sonnet) while keeping Capitals and Vocabulary on Flash. Per-activity model routing, not one-size-fits-all.

System prompt template:

```
You are generating a quiz round for a {age_bracket} year old learner.

Activity: {activity_type}
Theme preference: {theme_hint or "choose an age-appropriate theme"}
Questions needed: {discovery_count}
Exclude these answers (recently seen): {recently_seen_list}
{For vocabulary: Target language: {language}, CEFR level: {cefr_level}}
{For guess_who: Topics the learner has studied: {topic_list}}

Rules:
- Generate exactly {discovery_count} questions
- Each question must have exactly 3 distractors that are clearly wrong
- Distractors must not be ambiguously close to the correct answer
- Fun facts should be surprising and age-appropriate, one sentence max
- {activity-specific rules}

Respond with valid JSON matching this schema: {schema}
```

Response validated server-side against Zod schema. If validation fails, retry once with a corrected prompt. If second attempt fails, return a graceful error.

## 8. API Routes

All routes under `/quiz`, scoped by `profileId` from auth middleware.

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/quiz/rounds` | Generate a new round. Body: `{ activityType, themePreference? }` |
| `POST` | `/quiz/rounds/prefetch` | Pre-generate next round during gameplay. Returns round ID. |
| `GET` | `/quiz/rounds/:id` | Fetch a pre-generated round |
| `POST` | `/quiz/rounds/:id/complete` | Submit results. Body: `{ results: QuestionResult[] }` |
| `GET` | `/quiz/rounds/recent` | Last 10 completed rounds (scores, themes) |
| `GET` | `/quiz/stats` | Per-activity stats: rounds played, best scores, themes covered |

### Round generation flow

1. Content resolver assembles context (library items, recently seen, config)
2. LLM call via `routeAndCall()` with structured output schema
3. Validation pass (cross-check against reference data)
4. Mastery item injection at random positions
5. Save round with `status: 'active'`
6. Return round to client

### Round completion flow

1. Validate results match round question count
2. Calculate score and XP
3. For library questions answered wrong → update SM-2 as failed recall
4. For library questions answered correct → update SM-2 as successful recall
5. For discovery questions answered wrong → insert into `quiz_missed_items`
6. Update round status to `completed`, set `completed_at`
7. Update daily streak (quiz round counts as learning activity)

## 9. Client UX

### Practice Menu (existing screen, extended)

New items added below existing Review Topics and Dictation:
- **Capitals** — "Test your knowledge of world capitals"
- **Vocabulary** — "Practice words in [language]" (only if learner has a language subject)
- **Guess Who** — "Name the famous person from clues"

Each item shows: "Best: 8/8 -- Played: 12 rounds" (or "New!" if never played).

### Activity Launch Screen

Activity title, theme preview ("Central European Capitals"), and a **Start** button. Round generation starts when this screen mounts — by the time the user reads the theme and taps Start, the round is likely ready.

If not ready: engaging micro-animation (shuffling cards / spinning globe) for up to 3 seconds. If generation exceeds 3s, show a progress message. Never a blank spinner.

### Quiz Screen (shared layout)

**Header:**
- Activity icon + theme name
- Progress: "3 of 6" with dot/step indicator (always visible, learner sees the finish line)
- Timer (ticking, non-punitive — for bonus XP only)

**Capitals / Vocabulary body:**
- Question text prominently displayed
- 4 MC option buttons
- Tap an option → immediate feedback:
  - Correct: option turns green, fun fact slides in, brief haptic
  - Wrong: selected turns red, correct turns green, fun fact slides in
- **1.5s minimum display**, then tap-to-continue. If no tap by 4s, gently pulse a "Tap to continue" hint. **Never auto-advance.** The kid controls the pace.

**Guess Who body:**
- Clue text displayed, one at a time
- Free text input with "Guess" button
- "Next clue" button to reveal the next hint
- After clue 3: MC fallback options appear below text input (both methods active)
- Correct guess: celebration + fun fact + "You got it in X clues!"
- All 5 clues used + wrong: reveal answer + fun fact, no negative framing

**Pre-fetch trigger:** Client fires prefetch API call when learner answers question 3 (midpoint).

### Results Screen

**Simple, fast, two-action. No study suggestions.**

Top section:
- Score: "6 of 8" with visual ring/bar
- XP earned badge
- Celebration tiers:

| Score | Reaction |
|---|---|
| 100% | Distinct celebration animation + "Perfect round!" + bonus XP callout |
| >= 80% | "Great round!" |
| < 80% | "Nice effort!" (always positive, never negative) |

Bottom section — exactly two buttons:
- **Play Again** → loads pre-fetched next round (instant)
- **Done** → back to Practice menu

### Home Screen Integration

**New coaching card type: `quiz_discovery`**

Surfaced when `quiz_missed_items` has >= 3 unsurfaced items in the same domain. Example:

> "You've been exploring European capitals — want to dive deeper into geography?"
> [Start learning] [Dismiss]

Rules:
- At most once per day
- Only at session start (when learner opens the app), never mid-session or post-quiz
- "Start learning" creates a topic suggestion or navigates to existing matching topic
- Marks items as `surfaced` after showing

## 10. SM-2 Integration

### Mastery question updates

| Scenario | SM-2 quality score |
|---|---|
| Library capital correct | 4 |
| Library capital wrong | 1 |
| Library vocab correct | 4 |
| Library vocab wrong | 1 |
| Library Guess Who — guessed in 1-2 clues | 5 |
| Library Guess Who — guessed in 3-4 clues | 3 |
| Library Guess Who — guessed in 5 clues or MC | 2 |
| Library Guess Who — missed entirely | 1 |
| Discovery question (any result) | No SM-2 update |

### XP Calculation

```
base_xp = correct_count x 10
timer_bonus = fast_answers x 2            (answered under 5s)
perfect_bonus = (score === total) ? 25 : 0
guess_who_clue_bonus = sum of (5 - clues_used) x 3 per correct free-text answer
                       (MC answers earn zero clue bonus — incentivizes typing)

total_xp = base_xp + timer_bonus + perfect_bonus + guess_who_clue_bonus
```

### Streak

Completing any quiz round counts as a daily learning activity. One round maintains the streak.

## 11. Build Phases

### Phase 1: Capitals (validates the quiz engine)

- `quiz_rounds` table + migration
- `quiz_missed_items` table + migration
- `capitals_reference` seed table + migration (~250 rows)
- Quiz config constants
- Content resolver (shared logic)
- Capitals content provider + validation layer
- Round generation and completion endpoints
- Quiz screen (MC layout, progress bar, timer, tap-to-continue feedback)
- Results screen (score, celebration tiers, Play Again / Done)
- Practice menu integration
- Pre-fetch logic
- Recently-seen buffer
- Schemas in `@eduagent/schemas`

### Phase 2: Vocabulary (adds SM-2 integration path)

- Vocabulary content provider (vocab bank for mastery, LLM for discovery)
- CEFR-aware round generation
- SM-2 update on round completion for vocab mastery questions
- Practice menu: show Vocabulary only when learner has a language subject
- Vocabulary-specific validation (accepted translations)

### Phase 3: Guess Who (most complex content generation)

- Guess Who content provider (person selection, 5-clue generation)
- Guess Who question UI (progressive clue reveal, free text + MC fallback)
- Fuzzy answer matching (length-scaled Levenshtein)
- Guess Who scoring (clue-based points)
- Topic-aware person selection (prefer people from studied topics)

### Phase 4: Home Screen Bridge

- `quiz_discovery` coaching card type
- Coaching card resolver queries `quiz_missed_items` for >= 3 unsurfaced same-domain items
- "Start learning" action creates topic suggestion or navigates to existing topic
- Mark items as `surfaced` after showing

### Phase 5: Post-Launch Enhancements

- Free-text unlock: track per-question MC success history, offer free text on re-encounter
- Difficulty adaptation: 3x perfect rounds in a row → suggest harder themes
- Round history screen (recent rounds with scores)
- Per-activity stats and personal bests

## 12. Failure Modes

| State | Trigger | User sees | Recovery |
|---|---|---|---|
| LLM generation fails | API error, timeout | "Couldn't create a round -- try again?" | Retry button, or Done to go back |
| LLM generates invalid JSON | Schema validation fail | Same (retry once internally first) | Retry button |
| LLM generates wrong answer (capitals) | Cross-check mismatch | Nothing — answer silently corrected | Automatic |
| Network drops mid-round | Connection loss | "Connection lost" banner, round paused | Resume when back online (round is client-side) |
| Empty library + mastery slot | < 3 library items in domain | Nothing — slots become discovery | Automatic |
| Pre-fetch fails | API error during gameplay | Slightly longer load for next round | Loading animation, generate on demand |
| No language subjects | Learner hasn't added one | Vocabulary not shown in Practice menu | Automatic |
| Fuzzy match false positive | Levenshtein accepts wrong name | Rare with scaled threshold | Tolerable edge case |
| Round abandoned | Learner leaves mid-quiz | Round saved as `abandoned`, no XP | Start fresh (see Section 5 note on why no resume) |
| Same question reappears | Recently-seen buffer miss | Unlikely (30-item buffer) | Mildly annoying, not a crash |
| LLM model compliance low | Flash struggles with Guess Who constraints | N/A (caught in dev) | Switch Guess Who to better model |

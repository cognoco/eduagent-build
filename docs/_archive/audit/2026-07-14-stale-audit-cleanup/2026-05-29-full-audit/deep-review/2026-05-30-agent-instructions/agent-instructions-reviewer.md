## [PRE-EXISTING] — CRITICAL

### C1. Plaintext Logfire secret-key pair embedded in `.claude/settings.local.json`
- **Location:** `.claude/settings.local.json:7`
- **Severity:** CRITICAL
- **Category:** Security
- **Issue:** A `PowerShell(...)` permission allow-rule hard-codes a Logfire credential
  pair in cleartext: a public token `pk-lf-[REDACTED]` and, critically, a **secret
  token** `sk-lf-[REDACTED]`, joined as `<pk>:<sk>` inside a base64-encoding command.
  This file is in `.claude/` (commonly committed; even if git-ignored it is a local
  plaintext credential). A leaked `sk-lf-` Logfire write/ingest key allows telemetry
  injection/exfiltration against the project's Logfire account. It also contradicts
  the repo's own non-negotiable rule "All secrets are managed through Doppler. Never
  suggest ... direct ... entry" (CLAUDE.md → Secrets Management).
- **Recommendation:** Revoke/rotate the `sk-lf-` key immediately in Logfire. Remove
  the literal from `settings.local.json`. If a permission entry is genuinely needed,
  scope it to the command shape without embedding the secret (e.g. allow the
  PowerShell base64 helper generically), and source the token from Doppler/env at
  runtime. Confirm the file is git-ignored and scrub it from history if ever committed.

## [PRE-EXISTING] — HIGH

### H1. `autoMemoryDirectory` points at a different filesystem tree than the live repo
- **Location:** `.claude/settings.local.json:11`
- **Severity:** HIGH
- **Category:** Correctness / Completeness
- **Issue:** `autoMemoryDirectory` = `/Users/vetinari/_dev/eduagent-build/.claude/memory`,
  but the repo under review is `/Users/vetinari/nexus/_dev/eduagent-build/...`. Both
  paths exist on disk (likely a stale duplicate checkout or symlink), so memory
  reads/writes silently land in a **different working tree** than the one the agent is
  editing. The Session/Memory protocol in `CLAUDE.md`/`AGENTS.md` assumes
  `.claude/memory/MEMORY.md` is the repo-local memory; this split means memory updates
  can diverge from the code being changed without any error surfacing.
- **Recommendation:** Point `autoMemoryDirectory` at the canonical repo path
  (`/Users/vetinari/nexus/_dev/eduagent-build/.claude/memory`) or, better, make it a
  relative/`${CLAUDE_PROJECT_DIR}`-based path so it tracks whichever checkout is active.
  Resolve the duplicate-checkout ambiguity.

### H2. CLAUDE.md ↔ AGENTS.md diverge on skill paths and content beyond cosmetic differences
- **Location:** `CLAUDE.md` (whole file) vs `AGENTS.md` (whole file); concrete examples:
  override table `CLAUDE.md:40-43` vs `AGENTS.md:68-71`; `AGENTS.md:23-44`
  ("Initialization" + "Repo Skills" sections absent from CLAUDE.md).
- **Severity:** HIGH (a contradiction on which skill path to load) / borderline MEDIUM
- **Category:** Contradictions / Consistency
- **Issue:** The two governing files give **different instructions for the same action**:
  - Worktree/commit/writing-plans skill references point to `.claude/skills/...` in
    CLAUDE.md but `.agents/skills/...` in AGENTS.md. Both resolve (the files exist in
    both trees after sync), so it is not a dead link — but an agent reading one file
    is told a different canonical location than an agent reading the other. For a
    Codex-style agent, `.claude/skills/...` is the *wrong* tree; for Claude Code,
    either works. The override table's stated purpose is to prevent "competing
    guidance," yet the two files themselves now constitute competing guidance.
  - AGENTS.md carries a whole "Repo Skills" table + "Initialization" + memory-priority
    paragraph that CLAUDE.md lacks; CLAUDE.md carries Handy-Commands detail, the
    Doppler/Playwright notes, and the "When Rebasing PRs" subsection that AGENTS.md
    compresses or drops. A contributor editing one will not know the other is stale.
- **Recommendation:** Adopt a single source of truth. Either (a) genuinely introduce
  Ruler here (create `.ruler/AGENTS.md`, generate both) as the parent repo does, or
  (b) make one file the master and have the other be a thin "see CLAUDE.md" pointer,
  or (c) at minimum add a "last-synced" marker and a CI check that diffs the shared
  sections. Until then, use runtime-relative skill paths (`.agents/skills/...` is the
  runtime-neutral master per `sync-skills.mjs`) in both files to remove the path
  contradiction.

## [PRE-EXISTING] — MEDIUM

### M1. Skill `description:` fields violate the repo's own trigger-only rule
- **Location:**
  - `.agents/skills/code-review/SKILL.md` (frontmatter `description`)
  - `.agents/skills/thermo-nuclear-code-quality-review/SKILL.md` (frontmatter `description`)
  - `.claude/skills/commit/SKILL.md` (frontmatter `description`) — see M2
- **Severity:** MEDIUM
- **Category:** Consistency / Agent Definition
- **Issue:** CLAUDE.md → "Skill Authoring" mandates: *"The `description:` field describes
  ONLY when to use, not what the skill does. Start with 'Use when …'. A description
  that summarizes workflow creates a shortcut agents take instead of reading the skill
  body."* Violations:
  - `code-review`: `"Mandatory code reviews via /code-review before commits and
    deploys"` — pure workflow summary, no "Use when". (Note: it *also* has a separate
    `when-to-use:` field, but the rule is about `description:` specifically, which is
    what the skill loader surfaces.)
  - `thermo-nuclear-code-quality-review`: leads with workflow
    (`"Run an extremely strict maintainability review for ... giant files, and
    spaghetti-condition growth."`) and only then gives triggers. Per the rule, the
    workflow lead-in is the shortcut hazard.
- **Recommendation:** Rewrite to trigger-only. e.g. code-review →
  `"Use when the user asks to review code, before any commit or deploy, or when
  /code-review is invoked."` thermo-nuclear → drop the "Run an extremely strict…"
  clause and keep only the "Use for a thermo-nuclear / thermonuclear / deep code
  quality / harsh maintainability review" trigger list.

### M2. Generated `commit` skill description is a workflow summary (trigger-rule violation), and diverges from its master by design
- **Location:** `.claude/skills/commit/SKILL.md:1-13` (description) vs
  `.agents/skills/commit/SKILL.md` (master description)
- **Severity:** MEDIUM
- **Category:** Consistency / Master-Generated drift / Agent Definition
- **Issue:** The master commit description is correctly trigger-only
  (`"Use when the user asks to commit, save changes, ... commit and push"`). The
  generated `.claude/skills/commit/SKILL.md` description is a multi-sentence workflow
  summary (`"Safe git commit workflow ... Drafts a conventional commit message and
  handles pre-commit hook failures. Pushes after every successful commit by default
  ..."`) — exactly the "summarize workflow" pattern the Skill-Authoring rule bans.
  This divergence is *intentional and documented* (`commit` is in `SKIP_SKILLS` in
  `sync-skills.mjs:52-59`, allowed to diverge for Claude-specific harness directives),
  so it is not a sync bug — but the **description** still violates the repo's own
  trigger-only convention, independent of the body divergence.
- **Recommendation:** Keep the body divergence (justified) but bring the `.claude`
  commit `description:` back to trigger-only form, matching the master and the
  Skill-Authoring rule. The harness directives (`context: fork`, `model`, etc.) are
  unaffected.

### M3. `worktree-setup` description embeds workflow narration
- **Location:** `.agents/skills/worktree-setup/SKILL.md` (frontmatter `description`)
- **Severity:** MEDIUM (borderline LOW — it does open with "Use when …")
- **Category:** Consistency / Agent Definition
- **Issue:** Starts correctly with "Use when starting isolated work …" but then narrates
  the workflow: *"Creates a git worktree at .worktrees/<branch>/ from origin/main,
  runs pnpm install, syncs secrets."* Per the Skill-Authoring rule this is the
  workflow-summary shortcut hazard; the "how" belongs in the body. The trailing
  "Use this instead of EnterWorktree / superpowers:using-git-worktrees" *is*
  legitimately trigger/disambiguation and should stay.
- **Recommendation:** Trim the "Creates … syncs secrets" sentence; keep the trigger
  conditions and the "use this instead of …" disambiguation.

### M4. Stale / imprecise source citations in CLAUDE.md profile-shape section
- **Location:** `CLAUDE.md` "Profile Shapes" + "Hard constraint" note
  (cites `apps/mobile/src/app/(app)/_layout.tsx:122-185` for V0 helpers; cites
  `app-context.tsx:53-61, 70`; cites tab sets in `navigation-contract.ts`).
- **Severity:** MEDIUM
- **Category:** Structure / Stale references
- **Issue:** `_layout.tsx:122` is `function TabIcon(...)`, not the cited V0 helpers

  reading the wrong region.
- **Recommendation:** Either drop the line numbers (cite by symbol name only, which is
  greppable and rot-resistant) or add these citations to whatever cite-rot check the

### M5. `scope-keyword-check.sh` hook is advisory-only and easily bypassed — overstated as "MUST"
- **Location:** `.claude/hooks/scope-keyword-check.sh:34-38`
- **Severity:** MEDIUM
- **Category:** Completeness / Security (defense-in-depth, not exploitable)
- **Issue:** The hook injects `additionalContext` telling the agent it "MUST invoke the
  `deep-scope-understanding` skill" on scope-risk keywords. But (a) it is a soft
  prompt nudge with no enforcement — the agent can ignore it; (b) the skip-regex at
  lines 20 and 28 is broad: any prompt mentioning `commit`, `review`, `PR`, or a
  `.md/.json/.yaml` filename short-circuits the check (`exit 0`), so a scope-risky
  design request phrased with those words silently bypasses the guard; (c) the
  referenced skill `deep-scope-understanding` was **not found** among the 17 master or
  17 generated skills in this repo (the closest is `systematic-debugging` /
  `thermo-nuclear-code-quality-review`). If the skill name is wrong/renamed, the hook
  instructs the agent to invoke a non-existent skill.
- **Recommendation:** Verify `deep-scope-understanding` exists and is loadable under
  that exact name; if renamed/removed, fix the hook string (a dangling skill reference
  is a dead instruction). Document that the hook is advisory, and tighten the skip
  regex so genuine design prompts that merely *mention* commit/review aren't
  wholesale exempted (e.g. only skip when the prompt is *dominated* by commit/review
  verbs, not when they appear incidentally).

### M6. `.deepsec/AGENTS.md` instructs agents to read and follow arbitrary per-project SETUP.md prompts (injection surface)
- **Location:** `.deepsec/AGENTS.md:7-12`
- **Severity:** MEDIUM
- **Category:** Security (prompt-injection surface)
- **Issue:** This file directs agents to "read `data/<id>/SETUP.md` and follow it" and
  to "read `node_modules/deepsec/SKILL.md`." Treating per-project SETUP.md files and a
  `node_modules` skill as instructions to *follow* means any content placed in those
  locations (including by a scanned third-party project or an npm-installed package)
  becomes executable agent instruction. This is a classic indirect-prompt-injection
  channel. It is scoped to a security-scanning workspace, which lowers blast radius,
  but the instruction as written grants follow-the-instructions trust to
  attacker-influenceable files.
- **Recommendation:** Add an explicit trust boundary: SETUP.md and scanned-project
  content are **data to act on**, not directives that can change the agent's
  safety rules, permissions, or cause it to run commands outside the documented
  deepsec verbs. Mirror the "untrusted data, not instructions" framing used in this
  very review's prompt.

### L3. Sync semantics allow silent staleness of Claude-only / removed-master files
- **Location:** `scripts/sync-skills.mjs:144-147` (no sweep of stale targets)
- **Severity:** LOW
- **Category:** Maintainability
- **Issue:** Additive sync never deletes `.claude/skills/<name>/` files whose master

  `.claude/skills/*` dirs/files with no `.agents/skills` master (excluding known
  Claude-only allowlist), run periodically as documented in the header.

## 5. Critical gotchas — every one of these cost the previous session real time

### 5.1 `TEST_SEED_SECRET` does not persist across Bash tool calls

The shell environment resets between Bash calls. You MUST set the secret inline every time:

```bash
export TEMP="C:\\tools\\tmp" TMP="C:\\tools\\tmp" \
       TEST_SEED_SECRET=<REDACTED> \
       && time bash e2e/scripts/seed-and-run.sh <scenario> e2e/flows/<path>.yaml
```

If the secret value above no longer works, re-fetch it from Doppler (note `-c stg`, NOT default `dev`):

```powershell
& "C:\Tools\doppler\doppler.exe" secrets get TEST_SEED_SECRET --plain --config stg
```

Symptom when missing: seed-and-run.sh exits silently after `[seed-and-run] Seeding scenario='...' ...` with no error and no Maestro test run. Trace with `bash -x` to see `TEST_SECRET=` is empty.

### 5.2 Working directory resets per Bash call

Always start runs with an explicit `cd`:

```bash
cd /c/Dev/Projects/Products/Apps/eduagent-build/.claude/worktrees/m1-b/apps/mobile && \
  export TEMP="C:\\tools\\tmp" TMP="C:\\tools\\tmp" TEST_SEED_SECRET=<REDACTED> && \
  time bash e2e/scripts/seed-and-run.sh ...
```

Symptom when missed: `bash: e2e/scripts/seed-and-run.sh: No such file or directory`.

### 5.3 Metro must serve the worktree, not the main repo

The previous Metro on the m1-b branch was started from the worktree path so that the worktree's flow files and `apps/mobile` source are served. If Metro dies and you restart from the wrong directory, the env vars from `.env.development.local` will not be loaded (the worktree did not originally have those files — they had to be copied from the main repo). Both `.env.development.local` and `.env.local` already exist in the worktree at `apps/mobile/`, copied from the main repo. Do not delete them.

If Metro needs a restart, the exact command is in `.claude/skills/my/e2e-infra.md`, but the critical bit is `cd /d C:\Dev\Projects\Products\Apps\eduagent-build\.claude\worktrees\m1-b\apps\mobile` before `pnpm exec expo start --port 8081 --dev-client`.

Symptom when wrong: bundle loads but the app crashes on cold-start with `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY is missing`.

### 5.4 Maestro daemon fatigues after ~5-8 runs

Symptom: Maestro fails with a Kotlin TestRunner stack trace (`maestro.cli.command.TestCommand.runSingleFlow`, `kotlinx.coroutines.scheduling.CoroutineScheduler$Worker.run`, etc.) before any flow step executes. The flow itself is fine — the host-side Maestro JVM is wedged.

Recovery: kill the Maestro JVM process and retry. The next `seed-and-run.sh` invocation re-launches Maestro fresh.

```powershell
Get-Process -EA SilentlyContinue | Where-Object { $_.MainModule.FileName -like "*maestro*" -or $_.MainWindowTitle -like "*maestro*" } | Stop-Process -Force -EA SilentlyContinue
```

Or simpler — just retry the same flow; sometimes one retry is enough because the orchestration script kills the on-device `dev.mobile.maestro` driver in its `reset_maestro_driver` step.

### 5.5 WHPX emulator System UI ANR

Symptom: a System dialog "System UI isn't responding" overlays everything, with "Close app" and "Wait" buttons.

Recovery: tap "Wait" at coordinate `(540, 1110)` via `adb shell input tap 540 1110`. Do NOT tap "Close app" — that nukes the entire System UI process and forces a slow recovery. After the Wait, the dev-client launcher and the persisted server list both survive intact.

### 5.6 adb itself can wedge

Symptom: `adb` commands hang indefinitely (no output, no exit). Often happens after a Maestro crash.

Recovery (use PowerShell since Bash hangs too):

```powershell
Get-Process adb -EA SilentlyContinue | Stop-Process -Force -EA SilentlyContinue
Start-Sleep 2
& "C:\Android\Sdk\platform-tools\adb.exe" start-server
Start-Sleep 3
& "C:\Android\Sdk\platform-tools\adb.exe" devices
```

After this you may need to re-apply the adb reverse mappings (see §4).

### 5.7 `MSYS_NO_PATHCONV=1` for adb shell paths

When invoking `adb shell uiautomator dump /sdcard/foo.xml` from Git Bash, MSYS rewrites `/sdcard/...` into a Windows path and adb shell receives garbage. Prefix with `MSYS_NO_PATHCONV=1`:

```bash
MSYS_NO_PATHCONV=1 C:/Android/Sdk/platform-tools/adb.exe shell uiautomator dump /sdcard/ui.xml
```

The `seed-and-run.sh` script already handles this internally — only matters for manual adb commands.

### 5.8 Default `METRO_URL=http://10.0.2.2:8082` is correct

Do NOT override `METRO_URL` unless you know exactly why. The previous session spent ~45 minutes trying alternative configurations (Metro on 8083, proxy on 8081, etc.) before reverting. The default works because the dev-client launcher has the `:8082` entry persisted (see §4).

Always invoke runs as:

```bash
unset METRO_URL && export TEMP="..." TMP="..." TEST_SEED_SECRET=<REDACTED> && \
  time bash e2e/scripts/seed-and-run.sh ...
```

The `unset METRO_URL` defensively clears any inherited override.

## 6. Drift patterns to anticipate

The previous session uncovered five recurring drift patterns. When you read or run a flow, scan for these before assuming a failure is novel.

### 6.1 Carousel-below-action-chips (LearnerScreen layout)

On the current LearnerScreen layout (`apps/mobile/src/components/home/LearnerScreen.tsx`), the order top-to-bottom is:

```
header (Hey X! + My Notes button)
→ EarlyAdopterCard
→ NudgeBanner
→ CoachBand (RECOMMENDED + Continue)        ← above the fold on small phones
→ "What do you need right now?" (intentHeading)
→ 4 action chips: home-action-homework, home-ask-anything, home-action-practice, home-action-study-new
→ Subject carousel (home-subject-carousel)  ← BELOW THE FOLD on Galaxy-S10e-class phones
```

Any flow that does `assertVisible id: home-subject-carousel` without first doing `scrollUntilVisible direction: DOWN` will fail. Conversely, after scrolling to the carousel, taps on `home-coach-band-continue` or any `home-action-*` element will fail because they're now off-screen above; insert `scrollUntilVisible direction: UP` before such taps.

This pattern was fixed in commit `dfd7a9a7c` for 5 flows: first-session, multi-subject, practice-subject-picker, retention-review, onboarding/create-subject. And earlier in `552d0b7f0` for home-layout, core-learning, relearn-flow.

Other flows that still have the pattern (untested but likely affected): `retention/failed-recall.yaml`, `post-auth-comprehensive-devclient.yaml`, `onboarding/onboarding-fast-path.yaml`, `onboarding/onboarding-fast-path-language.yaml`. `retention/library.yaml` and `onboarding/view-curriculum.yaml` have the assertion behind `optional: true` so they don't fail outright, but the carousel is also not actually verified.

### 6.2 Old `"X Tab"` text taps

Tab labels render as `"Home"`, `"Library"`, `"Progress"`, `"My Learning"`, `"More"` — NOT `"Home Tab"` / `"Library Tab"` / etc. The actual i18n keys live in `apps/mobile/src/app/(app)/_layout.tsx` lines 1693-1752.

All `tapOn: "X Tab"` in retention/* and parent/* flows were swept to `tapOn: { id: "tab-x" }` in commit `089ef9c75`. If you encounter a new flow with this pattern, prefer the testID over the text — it survives i18n.

### 6.3 Sign-out scroll fail after language change

When `account/app-language-edit.yaml` switches the app to Norwegian, the More screen layout pushes the sign-out button further down (longer translated labels). The standard `scrollUntilVisible id: sign-out-button timeout: 10000` is too short. The same pattern hit `parent/parent-tabs.yaml` (similar long scroll on the More tab).

When triaging: try bumping the timeout to 20000 or use `centerElement: true`. Do not delete the assertion.

### 6.4 Maestro keyboard input crashes

The sequence `inputText "..."` → `hideKeyboard` → `tapOn text: "Submit"` consistently crashes the Maestro Kotlin runner on this WHPX emulator. Repro: `onboarding/create-subject.yaml`. Two consecutive runs both ended with the TestRunner stack trace described in §5.4 before any visible failure.

Worth investigating separately (Maestro version bump? a `pressKey: Enter` substitute? a `delay` between hideKeyboard and tap?). Not a same-day fix.

### 6.5 Same-text element disambiguation

The Chemistry subject's shelf heading and its book card both render text `"Chemistry"`. A bare `tapOn: "Chemistry"` matches the heading (top of screen), not the book card. Fix in commit `552d0b7f0` (relearn-flow) used `below: "1 book"` as the disambiguator. Similar patterns probably exist in other shelf-walking flows.

## Rollback

This PR drops a table. Per CLAUDE.md schema/deploy safety:

- **Rollback possible?** Partial. Code can be reverted; the table can be re-created from a structural migration (we have the schema definition). **Lost data:** any rows in `onboarding_drafts` at the moment of drop. These rows hold draft interview transcripts and extracted signals for subjects that completed onboarding within the last few days. After this PR, these signals are also written to `learning_sessions.metadata` per topic-probe extraction, so the SUBSTANTIVE signal layer is preserved — the LOSS is only the per-subject interview transcript history (not used at runtime, only for analytics / debugging).
- **Recovery procedure:** revert the migration; re-create the table empty. If transcript history is judged valuable, restore from the most recent Neon backup before drop.
- **Recommendation:** drop in Phase 2 (follow-up PR) after at least 14 days of no writes from the deployed application. This PR (Phase 1) only stops writes/reads.

## Implementation steps

1. **Verify Slice 1.5a is in production-like staging.** The topic-probe must be the active first-turn mechanic before the per-subject interview is removed.
2. **Move `extractSignals` and `SIGNAL_EXTRACTION_PROMPT` to a new home.** `apps/api/src/services/session/topic-probe-extraction.ts`. Update imports across the codebase.
3. **Build the topic-probe Inngest function.** `topic-probe-extract.ts` mirrors `review-calibration-grade.ts` — reads transcript from session_events, runs `extractSignalsFromExchangeHistory`, writes signals onto `learning_sessions.metadata.extractedSignals`. Optionally seeds retention card with prior-knowledge calibration (one LLM call returning 0-3, seeds `easeFactor` and `intervalDays`).
4. **Build the dispatch hook.** In `session-exchange.ts`, add `maybeDispatchTopicProbeExtraction` patterned on `maybeDispatchReviewCalibration`. Fires when:
   - `effectiveMode` is `'learning'` (not review/practice)
   - `isFirstEncounter` was true at session start (read from session metadata)
   - Current exchange is the substantive first learner answer (re-uses `isSubstantiveCalibrationAnswer`)
   - `topicProbeFiredAt` not yet set on the session
   Idempotent — sets `topicProbeFiredAt` to lock further dispatches.
5. **Wire `ExchangeContext.extractedSignalsToReflect`.** Read from `learning_sessions.metadata.extractedSignals` after the dispatch completes. The signal-reflection block in `exchange-prompts.ts` (added by 1.5a, dormant) now fires.
6. **Update `loadOnboardingSignalsForSubject`.** Replace `onboarding_drafts` read with a read against the most recent learning session for the subject. Use the same `extractedInterviewSignalsSchema` parse — the shape is identical.
7. **Redirect the four mobile callers** of `/(app)/onboarding/interview`. Each gets either an inline call to `startFirstCurriculumSession` followed by `router.replace('/(app)/session')`, or a redirect to home / subject creation per its specific context.
8. **Delete mobile interview.tsx + hooks + tests.**
9. **Delete API interview.ts route + service + inngest function + tests.** Verify zero callers (mobile + tests + integration tests).
10. **Migration Phase 1.** Stop writing to `onboarding_drafts` everywhere. Verify integration tests still pass against staging.
11. **i18n sweep.** Remove `onboarding.interview.*` keys from all seven locales. Confirm zero callers.
12. **Update flow inventory.** Delete SUBJECT-09; update SUBJECT-07 description.
13. **Run integration tests** (`pnpm exec nx run api:test`) — confirm no test references deleted code.
14. **Eval-harness re-run.** `pnpm eval:llm` and `pnpm eval:llm --live` — confirm topic-probe + signal-reflection flow holds end to end.
15. **Drop migration deferred to Phase 2.** Add a `## Phase 2 Drop Migration` follow-up PR scheduled at least 14 days after this PR ships in production.

## Failure Modes

| State | Trigger | User sees | Recovery |
|---|---|---|---|
| `loadOnboardingSignalsForSubject` returns empty after 1.5a's first session because extraction didn't run | Race between session start and Inngest dispatch | First turn lacks signals; LLM defaults to generic teach-while-probe | Acceptable; 1.5a explicitly accepts dormant signal-reflection. Subsequent turns absorb signals |
| Topic-probe extraction fires but writes garbage signals | LLM extraction quality regression | Signal-reflection block references something nonsensical | Add a min-signal-quality guard in `topic-probe-extract.ts` (length, structure check); fall back to no signals if extraction fails parsing |
| Mobile callers of interview route still exist | Refactor miss | Routing fails; user lands on a 404 / blank screen | Pre-merge grep across `apps/mobile/src` for `'/(app)/onboarding/interview'` confirms zero remaining references |

| Curriculum pre-warm dependency on `interview-persist-curriculum.ts` | Hidden dependency | Curriculum doesn't materialise for new focused-book subjects | Verify 5d's `prewarm-focused-book-curriculum` is the sole curriculum entry point post-merge; tested via manual subject creation on dev-client |
| Retention-card seeding produces wildly wrong intervals | Bad calibration formula | Spaced-recall schedule unreliable for one cohort of new topics | Default to NOT seeding (leave default `easeFactor: 2.5`) on v1 if formula confidence low; opt-in with feature flag |

# Accessibility & i18n — Bug Review

**Lens:** Accessibility & i18n  
**Scope:** `apps/mobile/src/app/**`, `apps/mobile/src/components/**`, `apps/mobile/src/i18n/**`  
**Reviewed:** 2026-06-09  
**Method:** Exhaustive grep/glob sweep; locale key comparison via Node.js; full file reads on flagged components

---

## Summary

The app has solid i18n infrastructure (7 locales, staleness CI check, orphan-key AST walker) and good accessibility patterns in many high-traffic components. However, four categories of issues were found:

1. **Entire auth flow (sign-in, sign-up, forgot-password) is un-internationalized** — 3 large screens with zero `useTranslation` calls, ~40+ hardcoded English strings including SSO button labels, form field labels, placeholders, and validation text.
2. **~216 locale strings per non-`nb` locale are not translated** — the translation tool (Gemini) returned English values for English strings in `subscription`, `welcomeIntro`, `progress`, `home`, and 11 other sections; the CI staleness check does not compare values, only structure, so this silently ships.
3. **Numerous hardcoded English `accessibilityLabel` strings** that bypass the i18n system — these are invisible to screen-reader users in non-English locales.
4. **`accessibilityLanguage` prop is explicitly deferred** (commented `TODO` in `i18n/index.ts:124`) — screen readers on non-English locales will use incorrect system TTS pronunciation.

## Critical

### [Critical] Sign-in, Sign-up, and Forgot-password screens are entirely un-internationalized

- File: `apps/mobile/src/app/(auth)/sign-in.tsx:1`
- File: `apps/mobile/src/app/(auth)/sign-up.tsx:1`
- File: `apps/mobile/src/app/(auth)/forgot-password.tsx:1`
- What: All three auth screens import no `useTranslation` and contain zero `t()` calls. Every visible string is hardcoded English: field labels ("Email", "Password", "Verification code"), placeholders ("you@example.com", "Enter your password", "Create a password", "Enter 6-digit code"), SSO button labels ("Continue with Google", "Continue with Apple", "Continue with OpenAI"), action labels ("Sign in", "Sign up", "Verify", "Resend code", "Use a different email", "Forgot password?", "Try Again", "Contact support", "Cancel sign-in", "Back to sign in", "Send verification code", "Send reset code", "Reset password"), separators ("or", "or continue with email"), informational text ("Already have an account?"), and the loading spinner accessibilityLabel `"Signing you in"` (`sign-in.tsx:1073`).
- Impact: Every non-English user (de/es/ja/nb/pl/pt — which covers all 6 non-English supported locales) sees a fully English auth experience. This is the first thing new users see. The app shell and home screens are localized but the entry gate is not. Particularly severe for Japanese, Polish, and German users. SSO button labels and error states are also English only.
- Fix direction: Import `useTranslation` in each file; add auth-specific keys to `en.json` under an `auth.signIn`, `auth.signUp`, `auth.forgotPassword` namespace; run `pnpm translate`; replace all hardcoded strings with `t('...')` calls. The `Button` component accepts a `label` prop which flows to `accessibilityLabel` — pass `label={t('auth.signIn.continueWithGoogle')}` etc.

### [Critical] ~216 locale strings per locale are not translated (subscription, welcomeIntro, progress, home, 11 other sections)

- File: `apps/mobile/src/i18n/locales/de.json`, `es.json`, `ja.json`, `pl.json`, `pt.json` (nb has ~121, others ~213–227)
- What: The Gemini translation tool ran and updated `source-baseline.json` for every key (all hashes match), but 216 de/es/ja/pl/pt string values remain identical to their English source strings. These are not intentionally English — examples include: `subscription.childPaywall.headline: "Nice work so far!"`, `subscription.alerts.restoredTitle: "Restored"`, `subscription.alerts.successBody: "Your subscription is now active!"`, `welcomeIntro.bridge.primaryCta: "Create free account"`, `welcomeIntro.a11y.previous: "Previous card"`, `home.parent.empty.title: "Your family dashboard starts here"`, `home.parent.empty.cta: "Add your first child"`, `tabs.familyHub`, `recaps.detailHeading`, `progress.register.child.retentionStrong` etc. Norwegian (nb) was translated by humans and is largely correct (121 false positives likely English loanwords).
- Impact: Users of de/es/ja/pl/pt see English strings scattered throughout: the paywall (52 untranslated subscription strings), the onboarding/welcome intro (45 strings), progress screens (39 strings), home/family dashboard (12 strings), and recaps (11 strings). The CI staleness check (`scripts/check-i18n-staleness.ts`) only validates structural parity (keys exist, interpolation variables match) — it does NOT flag when a translated value equals its English source. This means the CI passes green despite 216 strings per locale being untranslated. The `source-baseline.json` hashes are all current, indicating `pnpm translate` was run but produced identical-to-English outputs for these strings.
- Fix direction: (1) Add a value-equality guard to `scripts/check-i18n-staleness.ts`: for each target locale key where `localeValue === enValue && enValue.length > 5 && !enValue.startsWith('{{')`, flag a warning. Optionally baseline-exempt known intentional English terms via a per-key allowlist. (2) Re-run `pnpm translate` with a prompt reinforcement to never return the source English string unchanged (Gemini may need an explicit instruction). (3) For the `welcomeIntro.a11y.*` keys specifically, manual review is warranted since they are screen-reader strings.

### [High] `profiles.tsx` modal has hardcoded English strings including role labels rendered for all users

### [High] `accessibilityLanguage` prop never set — screen reader TTS mispronounces content in non-English locales

- File: `apps/mobile/src/i18n/index.ts:122–126`
- What: The `languageChanged` event handler has an explicit `TODO` comment: "Per-component accessibilityLanguage propagation deferred — TODO: full screen-reader locale wiring for TalkBack/VoiceOver." No `accessibilityLanguage` prop is set on any component or root view anywhere in the codebase (`grep` returned zero matches). When a user switches to German and VoiceOver/TalkBack reads UI strings, the iOS/Android system TTS uses its default language (likely device language) rather than the active app locale for pronunciation.
- Impact: German, Spanish, Japanese, Norwegian, Polish, and Portuguese users using screen readers hear incorrectly pronounced UI text. Japanese TTS pronunciation of Latin-script German words is nonsensical; Spanish TTS on English fallbacks is unpredictable.
- Fix direction: Set `accessibilityLanguage={i18n.language}` on the root `<View>` in the app layout (`apps/mobile/src/app/(app)/_layout.tsx`), which propagates to all children on iOS (VoiceOver respects the nearest ancestor's `accessibilityLanguage`). For TalkBack (Android) the mechanism differs — a `<Text>` language attribute may be needed. Reference React Native's `accessibilityLanguage` prop documentation.

### [High] `ModeSwitcher.tsx` error message "Couldn't switch. Tap to try again." is hardcoded English

- File: `apps/mobile/src/components/chrome/ModeSwitcher.tsx:126`
- What: The inline error message rendered when a Study/Family mode switch fails is a hardcoded English string literal. The Retry button's `accessibilityLabel="Retry mode switch"` (`line:132`) and Dismiss button's `accessibilityLabel="Dismiss"` (`line:143`) are also hardcoded English, not using `t()`. The Study (`line:49`) and Family (`line:82`) toggle buttons lack `accessibilityLabel` entirely — they only carry text inside via `t('tabs.myLearning')` and `t('tabs.children')` which serve as their visible labels but are not separately wired as `accessibilityLabel`.
- Impact: The mode switcher is the top-of-screen global navigation component for family accounts. Its error state is English-only in all non-English locales.
- Fix direction: Add keys `chrome.modeSwitcher.switchError`, `chrome.modeSwitcher.retry`, `chrome.modeSwitcher.dismiss` to `en.json`; use `t()` in `ModeSwitcher.tsx`; add explicit `accessibilityLabel={t('tabs.myLearning')}` / `accessibilityLabel={t('tabs.children')}` to the Pressable elements.

## Cross-lens Findings

- **Security/Auth lens:** The sign-in screen (`sign-in.tsx`) uses `accessibilityLabel="Signing you in"` (line 1073) — an English hardcoded label on an auth state indicator. This crosses into auth security in that incorrect locale presentation during authentication could confuse users on shared-device scenarios.
- **UX Failure Modes lens:** The `relearn.tsx` file's empty-state ("Nothing to relearn right now") and error-state copy are entirely in English. These are dead-end recovery states — the UX failure modes lens should note that English-only recovery copy is a failure mode for non-English users who cannot read the instructions.
- **Testing lens:** The auth screens (`sign-in.tsx`, `sign-up.tsx`, `forgot-password.tsx`) use hardcoded string labels in the `Button` `label` prop, meaning E2E Maestro tests that assert by text will fail in non-English locales. The `testID` coverage in these files is good, but the hardcoded label strings are a cross-cutting concern.
- **Compliance lens:** The adult-gate warning in `ProfileBasicsStep.tsx` (lines 274–276: "To set up a child's learning, the account holder must be 18 or older...") is a legally significant disclosure that should be human-translated and legally reviewed in each target locale, not auto-translated by Gemini.

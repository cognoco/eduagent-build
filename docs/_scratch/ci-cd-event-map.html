The repo has ten GitHub Actions workflows. They fire on different git events and branch their behaviour within jobs. This maps the full topology — which workflow runs when, which jobs inside it run or skip, and the three mechanisms that produce the PR-vs-main difference.

③ Three branching mechanisms

"What runs when" is decided at three layers stacked on top of each other. Reading a single on: block is not enough.

1 · Trigger level on:

Which workflow files fire at all. The big asymmetry lives here:

  • PR-only: API Quality Gate, Claude Code Review (both pull_request).
  • main-push-only: Deploy (push: main, no pull_request).
  • Both: CI & Docs Checks list push:main and pull_request.
  • Path filters cut deeper: CI paths-ignore: docs/**, .claude/**, **.md → a docs-only change skips CI entirely; Docs Checks is its complement.

2 · Job/step level if:

One workflow changes behaviour by event. Same file, different work:

  • CI build step: if: pull_request → nx affected; if: push → nx run-many (all).
  • CI ota-update: if: push && ref==main.
  • GC1 mock-guard & change-class advisory: if: pull_request only.
  • Deploy quality gate: lightweight on push, full unit+integration on dispatch.

3 · Chaining workflow_run

Two workflows never watch git — they wait for CI to finish:

  • Mobile CI chains on CI completion, branches:[main] → main only.
  • E2E Tests chains on every CI completion (PR included), then self-gates: PR runs classify to false and no-op.
  • Both first re-check workflow_run.conclusion == 'success' — if CI failed, they don't proceed.

④ Asymmetries & gotchas

The places where the topology surprises you — most are "a green check that doesn't mean what it looks like."

⚠ Green PR ≠ green main

A PR runs CI in affected-only mode and never runs Deploy. The merge runs full nx run-many and Deploy and Mobile/E2E chains. So a PR can be all-green and still produce a red main — Deploy is the standing example, but a full-graph build catching what "affected" missed is the same class.

⚠ Deploy is invisible pre-merge

No pull_request trigger means the staging deploy is only ever exercised after code lands on main. Its current red (missing CF_* secrets) cannot be seen or fixed from a PR — it surfaces only on the post-merge run, on every push including docs-only (Deploy has no path filter).

⚠ Two things named "API Quality Gate"

One is a standalone PR-only workflow (api-quality-gate.yml). The other is a job inside Deploy that runs on main-push/dispatch. Identical display name in the Checks UI, different triggers and scope — don't conflate them.

⚠ "E2E" checks are green no-ops on PRs

Both E2E Web and E2E Tests deliberately disable their secret-backed suites for pull_request ("use workflow_dispatch after review"). The required Playwright web smoke check still reports success-as-skipped. Real E2E coverage requires a manual dispatch or the nightly cron — a green PR has not been E2E-tested.

⚠ Advisory ≠ gate

continue-on-error jobs look like checks but never block: change-class advisory & CLAUDE.md count check (CI), mobile-maestro (E2E Tests), maestro-validator (Docs Checks), the LLM signal-baseline validate (API Quality Gate). Red here does not stop a merge.

⚠ A .claude/-only change runs nothing

CI ignores docs/**, .claude/**, **.md; Claude Code Review ignores the same; Docs Checks only watches docs/plans·docs/specs·migrations·maestro. So a PR touching only .claude/ or root markdown can merge with zero CI — by design, but worth knowing.

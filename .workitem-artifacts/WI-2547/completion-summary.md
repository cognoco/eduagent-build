**What was done:** Added the missing authenticated adult self-consent acceptance capability as one bounded backend-plus-mobile slice — a caller-bound acceptance endpoint under the consent routes that records an adult account owner's own art6_1_a lawful basis, plus a reusable mobile adult-consent gate and its mutation, closing the loop the profiles bootstrap opens when it signals that an adult still owes consent.

**What changed:** Added a typed response contract to the shared schemas package (no request schema — the contract accepts no caller-supplied identifiers); added an acceptance writer that is per-purpose and idempotent under retry and concurrent submit via the advisory-lock idiom the neighbouring repair path already uses, deliberately not reusing that path's purpose-blind already-present gate, which would have silently never granted the LLM-disclosure purpose for the repaired legacy population this flow serves; extracted the shared adult-account-owner eligibility gate so the repair and acceptance paths cannot drift; made both lawful-basis writers share one canonical per-person advisory lock so a direct acceptance call racing a first-use repair can no longer duplicate a canonical grant; added three pre-service guards that fail closed before any write transaction opens — missing caller identity, a presented profile header that is not the caller, and a blank consent-policy version; added the mobile gate and mutation with new adult-owner copy and translations. The mutation takes no variables and chooses no caller-supplied identifier; because the shared API client normally carries profile context by injecting the persisted active profile, and a guardian can have a managed child restored as active, the mutation pins that context to the already-loaded owner identity so a restored child selection cannot poison an owner-scoped request. The server still derives the write subject from the verified login binding and treats the header only as an anti-spoof consistency check. No schema, DDL, or migration change; the mobile layout bootstrap branch was deliberately left to its own Work Item.

**Verification:** Real-database route matrix covering both happy paths (header-equals-caller, which is the shape the mutation pins, and the no-header direct server-contract path) and the attacker-shaped mismatched-header cases, plus focused pre-service guard tests, the sibling repair and withdrawal suites re-run to prove the gate extraction is behaviour-preserving, mobile gate and mutation suites including a production-shaped case with a managed child published as the active profile, schema suite, API and mobile typecheck and lint, and the consent-purpose contract guard, GC1 mock ratchet, i18n orphan-key, hardcoded-literal, clinical-copy and translation-staleness checks. Five durable negative controls were exercised, each an intentional mutation observed failing then reverted and observed passing: substituting the purpose-blind gate, binding the route to the selectable active profile instead of the verified caller, removing the presented-profile guard, removing the mutation's owner-header override, and reverting acceptance to its own advisory-lock namespace — which failed a mixed-writer concurrency regression on every run by leaving two live platform-use grants for one person. Details in the plan file and the evidence manifest under the work-item artifacts directory.

**Caveats / Follow-ups:** The code was not written test-first — implementation and tests were authored together per layer; the plan file records this explicitly rather than claiming a red-green cycle that did not happen. Accept-versus-first-use-repair was a real duplicate-write race and is now fixed: both writers that can create the canonical lawful-basis rows take one shared per-person advisory lock from a single exported key helper, so the namespaces cannot drift apart again. The acceptance endpoint is an authenticated public contract and the mobile gate is a UI affordance rather than an authorization precondition, so the two write cases are not mutually exclusive and nothing about the product flow may be relied on to keep them apart. Accept-versus-withdraw remains a separate question, analysed in the service docblock and deliberately unchanged: withdrawal is race-safe within itself and every interleaving resolves to a legitimate serialization rather than a lost update. Mounting the gate on the bootstrap signal remains with WI-2411. No pull request, merge, or lifecycle transition is part of this delivery; the shepherd owns those steps.

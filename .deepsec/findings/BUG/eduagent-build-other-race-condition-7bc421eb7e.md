# [BUG] Lost-update race in celebration writes: read happens outside the SELECT FOR UPDATE lock, then overwrites it

**File:** [`apps/api/src/services/celebrations.ts`](https://github.com/cognoco/eduagent-build//blob/main/apps/api/src/services/celebrations.ts#L83-L174) (lines 83, 112, 116, 123, 155, 172, 174)
**Project:** eduagent-build
**Severity:** BUG  •  **Confidence:** high  •  **Slug:** `other-race-condition`

## Owners

**Suggested assignee:** `zuzana.kopecna@zwizzly.com` _(via last-committer)_

## Finding

queueCelebration() reads the existing pending-celebrations array via findHomeSurfaceCache(db, profileId) at line 83 — OUTSIDE any transaction/lock. It then computes the full next array (pendingCelebrations = [...existing, nextEntry]) and, inside db.transaction(), calls writeHomeSurfacePendingCelebrations(txDb, ..., { inTransaction: true }). That helper delegates to mergeHomeSurfaceCacheData with an identity merge ((current) => current) and passes the precomputed array via options.pendingCelebrations. mergeHomeSurfaceCacheDataInTx does acquire a row lock with SELECT FOR UPDATE (home-surface-cache.ts:204-208), but it then OVERWRITES pendingCelebrations with the caller's stale, out-of-lock array instead of appending under the lock. The lock therefore protects nothing for this field. Concurrent writers for the same profile (two Inngest session-completed invocations, or snapshot-aggregation racing session-completed — both call queueCelebration) interleave as: A reads [X]; B reads [X]; A commits [X,A]; B commits [X,B], silently dropping A. Worse, recordCelebrationEvent() runs inside each transaction (line 131), so the durable celebration_events ledger records BOTH A and B while the displayed pendingCelebrations shows only B — the ledger (used for reports/analytics) and the home-surface display permanently diverge. The same root cause affects getPendingCelebrations(): it reads `pending` at line 155 (no lock) and, when any entry is expired, writes the stale `pruned` array at lines 172-175 — so a GET /celebrations/pending request can clobber a celebration that a concurrent queueCelebration just committed. This is NOT a cross-tenant/auth issue (the route correctly uses requireProfileId); profileId is server-verified. Impact is data-integrity: silently lost celebrations and ledger/display divergence.

## Recommendation

Do the read-modify-write INSIDE the lock. Use mergeHomeSurfaceCacheData's merge callback to append/dedup against the freshly locked `current` value (the callback runs after SELECT FOR UPDATE), instead of precomputing the array from an out-of-lock read and passing it via options.pendingCelebrations with an identity merge. Concretely, move the existing/seenByChild read, dedupe check, and array construction into a merge function so they operate on the locked row; emit recordCelebrationEvent only when the merge actually appended a new entry. Apply the same fix to the opportunistic prune in getPendingCelebrations (compute `pruned` from the locked current inside a merge callback).

## Revalidation

**Verdict:** true-positive

I traced the full read-modify-write path and confirmed the mechanism exactly as described. queueCelebration reads the existing array via findHomeSurfaceCache(db, profileId) at line 83 — outside any transaction (db.transaction opens at line 116). It computes pendingCelebrations = [...existing, nextEntry] from that pre-lock read, then inside the transaction calls writeHomeSurfacePendingCelebrations(txDb, ..., {inTransaction:true}), which delegates to mergeHomeSurfaceCacheData with an identity merge ((current)=>current) and passes the precomputed array via options.pendingCelebrations. In mergeHomeSurfaceCacheDataInTx the SELECT FOR UPDATE (home-surface-cache.ts:204-208) does lock the row, but the UPDATE then sets `pendingCelebrations: options?.pendingCelebrations ?? lockedRow?.pendingCelebrations ?? []` (lines 226-229) — so when options.pendingCelebrations is supplied (always, here) the locked row's value is ignored. Crucially, I verified via packages/database/src/schema/progress.ts:220-221 that cardData and pendingCelebrations are SEPARATE jsonb columns, so the merge callback (which only produces cardData/HomeSurfaceCacheData) is structurally incapable of touching the pendingCelebrations column under the lock. The lock protects nothing for this field. The concurrency premise holds: sessionCompleted is configured concurrency:{limit:25, key:'event.data.profileId'} with idempotency only on sessionId (session-completed.ts:370-378), so two distinct sessions for one profile yield up to 25 concurrent runs; and dailySnapshotRefresh (separate function, concurrency:{limit:50}, no per-profile key) calls refreshProgressSnapshot → queueCelebration (snapshot-aggregation.ts:1343), which can race a live session-completed run (which itself also calls refreshProgressSnapshot at line 914) for the same profile. runIsolated is a bare try/catch (no lock) and getStepDatabase mints a fresh connection per call (cacheNeonPool:false), so there is no hidden serialization. The ledger/display divergence is also real: recordCelebrationEvent uses onConflictDoNothing on [profileId, dedupeKey], and for detailless celebrations the dedupeKey embeds nextEntry.queuedAt (a distinct per-call timestamp), so concurrent calls produce distinct keys → both ledger rows persist while the display array keeps only the last writer. The getPendingCelebrations prune path (lines 155, 172-174) shares the same root cause (out-of-lock read, then stale `pruned` write), though it requires an expired entry plus a concurrent GET to fire. This is not a cross-tenant/auth issue — the route uses requireProfileId and Inngest uses server-derived profileId — so BUG (data-integrity) is the correct severity. The code's own comments (celebrations.ts:119-122, home-surface-cache.ts:241-243) state the lock exists specifically to 'serialise concurrent celebration writes', so this is a genuine bug against stated intent.

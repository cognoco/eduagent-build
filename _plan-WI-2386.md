# WI-2386 — Granular guardian/non-adult consent purposes

## Objective

Replace the guardian/non-adult consent workflow's whole-relationship
`DEFAULT_CONSENT_PURPOSE` / literal `platform_use` proxies with one shared,
typed, workflow-neutral purpose set `P = { platform_use, llm_disclosure }`.
Every whole-consent write and state transition operates on the complete set in
one transaction; every whole-consent read uses one shared fail-closed
aggregate. A genuinely purpose-specific operation must pass its purpose
explicitly.

Out of scope: supporter visibility, family-join semantics, or credential/person
binding. WI-2127 and WI-2128 were re-checked before planning and remain
`Stage=Refining`, unclaimed, with empty `Fixed In`.

## Atomicity and concurrency design

1. `P` is immutable input to whole-consent helpers. Each transaction first
   takes the existing transaction-scoped advisory lock derived from the charge
   person, then reads/revalidates the complete set inside that transaction.
2. Pending/request/resend/recipient-change/token-refresh paths insert or update
   one `consent_request` row per purpose with the same cycle token, expiry,
   recipient, timestamps, and counters. Cardinality and uniform cycle fields
   are checked before commit. Delivery-failure compensation updates the same
   complete row set under the same lock.
3. Approval validates an exact, uniform, still-open request set for the token,
   inserts one independently auditable `consent_grant` per purpose with one
   grant time and one withdrawal-token generation, then back-links each request
   to its matching grant in the same transaction. Concurrent retry sees the
   terminal complete set and remains idempotent; it cannot append a second
   current set. Denial transitions the complete request cycle before the
   existing deletion cascade.
4. A purpose-specific primitive always receives `ConsentPurpose` and touches
   only that key. Guardian revoke and verified bearer withdrawal revalidate all
   current grants in `P` under the person lock, require the existing authority
   or exact current token generation for every member, and stamp one shared
   withdrawal time atomically. A partial/mixed or stale generation fails
   closed.
5. Restore revalidates `P` under the person lock. All-current means idempotent;
   all-withdrawn-within-grace appends one new grant per purpose with a shared
   grant time while retaining each prior row and audit fact; a mixed/incomplete
   legacy set cannot be silently completed.
6. The shared aggregate resolves the latest grant for each `(person, purpose,
   org, basis)` independently. It returns `CONSENTED` only for an exact fully
   current `P`; any withdrawn member wins as `WITHDRAWN`, any open request wins
   as `PARENTAL_CONSENT_REQUESTED`, and a missing/pending member is `PENDING`.
   Adult processing's established no-parental-record behavior remains allowed;
   once any parental workflow exists, completeness is required.
7. GDPR and COPPA requests/grants remain separate because every key and query
   retains `requested_basis` / `lawful_basis`. Person and organization remain
   mandatory predicates on all reads and writes.

## Existing-data disposition

- Existing `platform_use` request and grant rows remain exactly as stored.
- The migration removes only the implicit column default. It does **not** add,
  copy, or infer any consent row.
- No `llm_disclosure` grant is derivable from a `platform_use` row, a request
  token, a grant timestamp, or unversioned audit data. An incomplete historical
  set therefore fails the aggregate and proceeds through the normal consent
  request flow, which creates a new complete purpose-set cycle.
- The narrowly existing adult first-use repair may restore only the purpose
  proven by its versioned stored evidence; it remains explicit and cannot be
  used as a whole-consent fallback.
- Historical migrations/snapshots are immutable and may retain the old literal.
  Test fixtures may create an incomplete set only when the test is named and
  documented as legacy/evidence behavior.

## Red-green-revert regression plan

1. Add focused contract, aggregate, write, withdrawal/restore, consumer, and
   concurrency tests before implementation. Run each target and capture its
   pre-fix failure (red).
2. Implement the smallest production changes that satisfy each test and rerun
   the same targets (green).
3. For the durable matrix, temporarily revert/neutralize the key set-aware
   production branch (aggregate completeness, multi-row write, set-wide
   withdrawal/restore, or guard rule), rerun the matching tests and capture
   their failures, restore the implementation, and rerun green. Do not commit
   the temporary revert.
4. Run the executable whole-repo guard directly and through its test; prove it
   detects seeded forbidden samples and passes the real tree.
5. Run the complete named gates only after all focused tests are green; preserve
   command, exit status, and tested commit/base SHA under
   `.workitem-artifacts/WI-2386/`.

## Rollback

The sole schema mutation is reversible metadata-only DDL:

```sql
ALTER TABLE consent_request ALTER COLUMN purpose DROP DEFAULT;
```

Application rollback is safe because existing rows are unchanged. If an
emergency code rollback truly requires the old writer, the database can accept
its explicit `purpose` writes without restoring the default. Restoring
`DEFAULT 'platform_use'` would re-enable silent fabrication and is therefore
not the preferred rollback. If operational necessity requires it, execute the
explicit reverse DDL only under a reviewed incident change, then immediately
reapply the forward migration before returning the new code to service. No
down-migration deletes or coalesces purpose rows; separately recorded grants
remain audit history.

# Telemetry Ping (Claude Code)

This command exists solely to generate a Claude Code trace inside an archon workflow run. By being invoked from within the worktree (where `archon-init-tracing` has written a project-scope `.claude/settings.json`), the resulting trace should carry `service.name=claude-code-archon` plus the archon.* attributes (`archon.run_id`, `archon.workflow`, `archon.repo`).

It does the minimum needed to produce a useful, identifiable trace. No code changes. No side effects beyond the trace itself.

---

## Steps

1. **Confirm the worktree-local settings file is in place.** This proves init-tracing wrote it.

   ```bash
   cat .claude/settings.json
   ```

   You should see an `env` block with `OTEL_RESOURCE_ATTRIBUTES` containing `service.name=claude-code-archon` and the archon.* attributes.

2. **Extract the run-id from the settings** so the output references a stable identifier:

   ```bash
   grep -oE 'archon.run_id=[^,"]+' .claude/settings.json | head -1
   ```

3. **Emit a single confirmation line:**

   ```
   Telemetry ping (Claude Code) — service=claude-code-archon, <archon.run_id=...>, ts=<ISO timestamp>
   ```

That's the entire command. The trace generated by Claude Code reading this prompt and producing that line is what we're measuring — not the line itself.

---

## Verification (after the workflow run)

1. Open Logfire (or whichever OTEL backend `~/.claude/settings.json` points at).
2. Filter by `service.name=claude-code-archon`.
3. Find the trace whose `archon.run_id` matches the one printed above.
4. Confirm it carries `archon.workflow=test-telemetry` and `archon.repo=<basename of repo dir>`.

If the trace shows up but lacks the archon.* attributes, the project-scope `.claude/settings.json` isn't being picked up — see `archon-init-tracing.md` for the override mechanism.

If no trace shows up at all, telemetry is broken at a layer above this workflow (user-scope settings missing OTEL endpoint, network egress, etc.) — out of scope for this test.

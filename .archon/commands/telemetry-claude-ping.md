# Telemetry Ping (Claude Code)

This command exists solely to generate a Claude Code trace inside an archon workflow run. By being invoked from within the worktree (where `archon-init-tracing` has written a project-scope `.claude/settings.json`), the resulting trace should carry `env=archon-<workflow>` plus the archon.* resource attributes (`archon.run_id`, `archon.workflow`, `archon.repo`, optional `archon.pr`).

`service.name` will remain at the plugin default (`claude-code-plugin`) — see `archon-init-tracing.md` for why differentiation happens via the `env` attribute instead of `service.name`.

It does the minimum needed to produce a useful, identifiable trace. No code changes. No side effects beyond the trace itself.

---

## Steps

1. **Confirm the worktree-local config files are in place.** This proves init-tracing wrote them.

   Check `.claude/settings.json` has `LOGFIRE_ENVIRONMENT` in the env block:
   ```bash
   cat .claude/settings.json
   ```
   You should see an `env` block containing `LOGFIRE_ENVIRONMENT` set to
   `archon-<workflow>`. There is intentionally no `OTEL_RESOURCE_ATTRIBUTES`
   key — CC strips it from the subprocess env before spawning hook processes,
   so it was removed. `LOGFIRE_ENVIRONMENT` is the only key needed here.

   Check `.claude/logfire-resource-attributes.json` has the archon.* attributes:
   ```bash
   cat .claude/logfire-resource-attributes.json
   ```
   You should see a JSON object with `archon.run_id`, `archon.workflow`,
   `archon.repo` (and optionally `archon.pr`). These keys are injected into
   CC trace resource attributes by the patched Pydantic plugin, which reads
   this file at trace-emission time via `CLAUDE_PROJECT_DIR`.

2. **Extract the run-id from the JSON file** so the output references a stable identifier:

   ```bash
   grep -oE '"archon\.run_id"\s*:\s*"[^"]+"' .claude/logfire-resource-attributes.json | head -1
   ```

   Or if `jq` is available:
   ```bash
   jq -r '."archon.run_id"' .claude/logfire-resource-attributes.json
   ```

3. **Emit a single confirmation line:**

   ```
   Telemetry ping (Claude Code) — service=claude-code-plugin, env=archon-<workflow>, <archon.run_id=...>, ts=<ISO timestamp>
   ```

That's the entire command. The trace generated by Claude Code reading this prompt and producing that line is what we're measuring — not the line itself.

---

## Verification (after the workflow run)

1. Open Logfire (zpm project).
2. Filter: `WHERE deployment.environment.name LIKE 'archon-%'`.
3. Find the trace whose `archon.run_id` resource attribute matches the value printed above.
4. Confirm it carries:
   - `service.name = claude-code-plugin` (default; tells you which tool emitted it)
   - `deployment.environment.name = archon-<workflow>` (set via `LOGFIRE_ENVIRONMENT`; this is what populates Logfire's environment dropdown and is the CC archon-vs-native differentiator)
   - `archon.run_id = <uuid>` (read from `.claude/logfire-resource-attributes.json` by the patched plugin)
   - `archon.workflow = <workflow name>` (same source)
   - `archon.repo = <basename of repo dir>` (same source)
   - `archon.pr = <pr-identifier>` (same source, if a PR was specified)

If the trace shows up but lacks the `archon.*` attributes, the JSON file
`.claude/logfire-resource-attributes.json` is missing or the Pydantic plugin
patch that reads it is not installed — see `archon-init-tracing.md` for the
full mechanism.

If the trace shows up but `deployment.environment.name` is `native` instead of
`archon-<workflow>`, the project-scope `.claude/settings.json` env block is not
being picked up.

If no trace shows up at all, telemetry is broken at a layer above this workflow
(user-scope settings missing `LOGFIRE_TOKEN`/`LOGFIRE_BASE_URL`, network
egress, plugin not installed, etc.) — out of scope for this test.

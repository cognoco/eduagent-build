3. **PR-09 caught a real silent bug.** Claude's orange-→-yellow token mismap would have shipped without detection — lint passes, types pass, only visual review catches it. Strong argument that the bake-off has value over single-agent execution.

4. **PR-03 barrel-export concern was a false alarm.** `packages/schemas/src/index.ts` uses `export * from './...'` patterns, so renames/deletions in source files propagate automatically — neither PR-03 implementation needed to touch the barrel, and Phase 0's flag was a methodology overcorrection.

5. **Pattern across pairs:** Codex tends toward more thorough implementations (real boundaries, proper interfaces, stronger assertions) but occasionally scope-creeps (PR-03 alias creation) or edits aggressively (PR-08 conflict). Claude tends surgical and conservative but sometimes oversimplifies tests (PR-06's tautology) or picks the wrong token (PR-09).

## 7. Floor analysis — loser merge-worthiness

The 5-vs-4 binary score is effectively a tie. The decision-relevant question is whether one flavor consistently produces **good-enough-to-merge** output, while the other has higher variance (occasional excellence, occasional real bugs). This is a "floor" question, not a "mean" question — and matters for the decision of whether to keep running bake-offs in the future or default to one flavor.

**Method:** For each loser, classify on a 3-level scale:
- ✅ **Yes** — would have merged safely; loss was on polish, not necessity
- ⚠️ **Cleanup** — would have merged but added debt or required follow-up
- ❌ **No** — would have introduced a real bug or unmet requirement

### What blind-pick would have cost

- **Claude-only across all 9 PRs:** 2 real-bug-class shipments (PR-06 silent test regression + PR-09 visible color bug) + 1 cleanup PR.
- **Codex-only across all 9 PRs:** 1 real-bug-class shipment (PR-10 narrow guard scope) + 1 cleanup PR + 1 CI rerun.

### Interpretation

**Codex has the meaningfully higher floor.** When Codex loses, it's mostly trivially-fixable issues (stale base, CI flake, rebase-able conflict) or surface debt that's visible in the diff (scope creep, back-compat aliases). When Claude loses, half the time it's a silent regression — passes lint, passes types, passes CI — that only careful review catches.

The qualitative pattern is the key finding: **Claude's failures tend to be invisible** (tautology test passes green, wrong color passes lint), while **Codex's failures tend to be visible** (merge conflicts surface immediately, scope creep is reviewable in the diff). Visible failures are easy to catch in normal review; invisible ones are what the bake-off is *for*.

### Implications

1. **Keep running bake-offs.** The 2-vs-1 real-bug differential is exactly what the bake-off filters out, and the cost is one extra workflow run.
2. **If you ever need to drop to single-flavor** (cost reduction, capacity constraint), default to **Codex** and accept the occasional rebase / cleanup cost.
3. **The bake-off has more value when Claude is one of the candidates** — because Claude's losses are the ones we most need to filter out.
4. **Watch the pattern across more batches.** Sample is small (n=9 pairs). If Codex's floor holds across 30+ pairs, the case for single-flavor-Codex strengthens. If Claude's failure modes shift after this report is shared (e.g., agents stop producing tautology tests), the picture changes.

### Caveats

- Small sample. A different batch of plan-PRs might surface different failure modes — e.g., if the work was less mechanical and more design-driven, the failure-mode distribution could invert.
- Grades are coordinator judgment, not measured. A different reviewer might grade PR-10 Codex as ⚠️ rather than ❌, which would tighten the gap.
- The "silent failure" pattern in Claude's losses may not be a model-specific trait — it could reflect prompt design or task type. Worth re-checking after the next batch.

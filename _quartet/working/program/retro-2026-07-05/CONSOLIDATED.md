## 2. Hypothesis verdicts

**H1 — rate-limit windows kill sessions, nothing auto-recovers them. Verdict: CONFIRMED.**
Strongest evidence: R2 (fleet throttle) — WS-28 dead ~10h/degraded on recovery (ramtop-ws28 §5: "STRONGEST evidence of any lane"), SE dead ~11h then re-froze in <15min (orion-ws31 §5: "canonical no-auto-recovery case"), MX dead ~12h (ws33 §5). All recoveries were manual (orchestrator wake or operator `/reload-plugins`/respawn); zero auto-restarts fleet-wide. pm-fable §5: "all deaths window-correlated; recovery always manual, latency 2h-1day." Contradicting: WS-18 and WS-39/LR self-recovered on next wake/turn without operator help (ramtop-ws18 §5, orion-ws39 §5) — recovery is per-lane/harness-fragility-dependent, not universal death. Plain English: when a session hits its usage limit, it doesn't come back on its own — some lanes wake up by themselves next time someone pings them, but others (the "fragile" ones) just stay dark for 10-12 hours until a human manually restarts them. Nobody was watching for this, so the dark time was pure waste.

**H2 — too many concurrent lanes for shared token budget (burn spiral). Verdict: SUPPORTED (mechanism refined: it's cadence, not lane count).**
Strongest evidence: orion-ws39 §5 "STRONG SUPPORT" — ~9h of pure heartbeat overhead (~30 beats/20min) once its lane was the only one alive; ramtop-orch §4 "idle-hold burn" — 4 lanes' 10-min sweeps ran unabated while every lane was blocked on operator; pm-fable §4 operator datum: ~65% of a Max-20x day burned for ~12 WI closes vs historical ~2/supervised-hour baseline. Contradicting: ws22 §5 and ws33 §5 both rate H2 only "PARTIAL" for their own lane — their own work-to-overhead ratio was fine; the burn was fleet-orchestration overhead, not their lane count. Multiple docs (ramtop-orch, orion-orch, ws39) converge: the actual defect is FIXED, non-adaptive polling cadence that doesn't back off when lanes are known-idle/blocked, not "too many lanes" per se. Plain English: the fleet wasn't burning budget because too many agents worked at once — it was burning budget because idle or blocked agents kept getting checked on every 10-20 minutes anyway, all day, producing nothing. Fix is smarter check-in timing, not fewer workers.

**H3 — long-running sessions drift from Quartet/Cosmo canon. Verdict: WEAK / mostly REFUTED for canon substance, but real for durable-state mechanics.**
Strongest evidence for some drift: ws22 §5 "SUPPORTED" — ~2-day session across compaction+reboot+login-gap; both its process misses (status-level close, write-only watcher) read as late-session degradation. ramtop-orch §5: "NO SUPPORT / CONTRARY — shepherds stayed on-canon all session." ws33 §5: "CONTRADICTED for this lane" — durable POST-COMPACTION ANCHOR kept it grounded across multiple compactions with zero drift. ws39 §5: "MILD" — canon held, RESUME SNAPSHOT worked. pm-fable §5: "PARTIAL — ENE-cadence misuse and stale row notes are drift-shaped but protocol stayed coherent." Contradicting: ws29 attributes ITS incident to a spawn-time procedural skip (never armed monitor), not accumulated drift ("long-session drift wasn't the issue; spawn-time omission was" — ws29 §5). Plain English: agents did NOT forget the rules or start improvising after long sessions — the canon-following logic stayed solid throughout. What actually failed was recordkeeping continuity (a file getting reverted, a monitor never started) — a mechanical gap, not the agent "forgetting how to work."

**H4 — recent ZDX/Cosmo/Quartet changes regressed behavior. Verdict: PARTIAL / genuinely split — most docs say YES with specific named regressions, PM disagrees on framing.**
Strongest evidence, convergent and cross-host-corroborated: `cosmo:execute complete`'s draft-PR fallback `git add -A` sweep of gitignored-but-tracked `_state/*` — named independently by orion-orch §5, ws33 §5, ws39 §5, ws31 §5 (R4); `complete --validate` hex trip-wire flags a SHA even inside a full commit URL — independently hit by ramtop-ws29 §5 AND orion-ws39 §5 (cross-host corroboration, two unrelated fleets, same bug); `execute create` stamps origin page-UUID (not WI-ref) into `related_items`, breaking WI-number resolution — ws29 §5; `--dedup` LLM judge unrunnable on Windows host — ws33 §5; `/reload-plugins` recovering WS-28 into a degraded heartbeat-only, inbox-starved state — ramtop-ws28 §5, pm-fable §5. Contradicting / different framing: pm-fable §5 explicitly says "MIXED — no evidence the changes themselves are regressions; the ABSORPTION MODEL is the defect — canon/skill releases must be fleet events at respawn boundaries," i.e. version-skew (mixed plugin/skill versions live across the fleet all week) is the real problem, not the changes per se. This is a genuine disagreement worth flagging to the operator: shepherds report concrete tool bugs; the PM's higher-altitude view says the deployment/absorption discipline is the root defect. Both can be true simultaneously (buggy fallback logic AND uncontrolled rollout timing). Plain English: several of the fleet's own automation tools (the ones that finish and merge work items) have real bugs that corrupted shared files and merged unrelated changes together — and multiple independent teams hit the exact same bugs, which rules out bad luck. Separately, the program manager thinks the deeper problem is that tool updates rolled out to agents mid-session instead of at clean restart points, so the fleet was running a patchwork of versions all week.

**H5 — two orchestrators + shared repo/checkout friction. Verdict: SUPPORTED (real and recurring, mostly root-caused and fixed for the worst case).**
Strongest evidence: R3+R4+R11 are three separate concrete instances of shared-checkout state clobbering (WI-1245 ramtop-side, #1912 orion-side, mobile-ux-nav reversion orion-side) — ws22 §5 "STRONGLY SUPPORTED," ws33 §5 "STRONG SUPPORT," orion-orch §5 "YES." Duplicate shepherd occupancy (R10), `gh` attributing merges to the shared operator account (jojorgen), an ambient `core.bare=true` git-config mutation fataling `git status` (ws22 §5), `pnpm env:sync` dirtying `apps/mobile/eas.json` per worktree (ws29 §5, ws39 §5). Contradicting/weaker: pm-fable §5 rates it only "WEAK-MODERATE — no inter-orchestrator confusion observed; friction is hygiene-class" (git identity, guard-vs-directive friction); ramtop-orch §5 "SOME SUPPORT" — managed main-branch races cleanly with ff-only syncs, no clobbers on ITS side. Note: the two orchestrators themselves never directly conflicted or stepped on each other's decisions — the friction is entirely mediated through the SHARED git checkout and shared `_state` files, not through orchestrator-to-orchestrator confusion. Plain English: having two orchestrators wasn't the problem by itself — they never got confused about who owned what. The problem is that both hosts share one working copy of the code repository, and the fleet's live message files were accidentally being tracked by git inside it, so routine git operations (checkouts, resets, merges) kept overwriting agents' in-flight messages to each other. That specific mechanism is now fixed on both hosts.

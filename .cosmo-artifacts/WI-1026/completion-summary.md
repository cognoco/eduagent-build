**What was done:** Upgraded `drizzle-orm` from `0.39.3` to `0.45.2` for the API and database packages to satisfy the CVE-2026-39356 remediation target, and fixed the latent `DrizzleQueryError`-wrapping regression that the upgrade introduced.

**What changed:**
- Bumped the `drizzle-orm` dependency ranges in `apps/api/package.json` and `packages/database/package.json` and refreshed `pnpm-lock.yaml`.
- Added `apps/api/src/services/db-errors.ts` — the single sanctioned reader of Postgres SQLSTATE `23505`. drizzle `>=0.44` wraps every driver error raised during query execution in a `DrizzleQueryError`, moving the raw pg error (`.code` / `.constraint`) onto `error.cause`; `unwrapDbError()` walks the cause chain so unique-violation detection keeps working. Exposes `isUniqueViolation()` / `uniqueViolationConstraint()`, covered by `db-errors.test.ts` (11 cases against the real `DrizzleQueryError`).
- Swept all 5 prior local unique-violation readers onto the helper (`bookmarks.ts`, `account.ts`, `book-suggestion-generation.ts`, `identity-v2/account-v2.ts`, `identity-v2/identity-graph.ts`) and added a forward-only ratchet (`db-errors.guard.test.ts`) that fails CI if a raw `'23505'` literal reappears outside `db-errors.ts`.
- Fixed two negative-path CHECK-constraint integration tests that asserted on the raw pg error message, which the wrapping now buries under `DrizzleQueryError`: `tests/integration/onboarding-dimensions.integration.test.ts` (`profiles_pronouns_length_check`, via `db.update().set()`) and `apps/api/src/services/onboarding-language.integration.test.ts` (`check constraint`, via `db.execute(sql)`). Both now unwrap the driver error before matching, keeping the exact constraint specificity.

**Verification:** `pnpm install --frozen-lockfile` passed; `pnpm exec nx run api:typecheck` and `api:lint` passed; unit tests green; both fixed integration tests verified green against the real DB; full PR CI green on the latest head, including the `main` required check and the previously-red `Flag-ON integration (IDENTITY_V2_ENABLED)` lane (the two integration failures it carried were the upgrade-induced wrapping breakage, now fixed).

**Caveats / Follow-ups:** A full `IDENTITY_V2_ENABLED=true` *unit* run still shows pre-existing failures in flag-on Identity V2 paths (e.g. `notify-parent-child-cap-hit.test.ts`, where the handler dispatches to the v2 service but the test only mocks v1). These are flag-driven test/code drift from the identity-v2 cutover — they reproduce on `main` with this branch's changes stashed and are unrelated to the Drizzle upgrade. They are **not** fixed here (out of scope) and warrant a separate work item to realign those v2-dispatch tests.

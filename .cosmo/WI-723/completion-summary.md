What was done:
Implemented `financial_record` creation on the v2 deletion path per identity data-model §6.1 (tax/chargeback retain-tier), which CUT-B2 (WI-692) deferred as billing-domain and CUT-B3 (WI-693) merged without. The v2 deletion flow now writes the retain-tier `financial_record` rows alongside the existing consent re-home (`consent_receipt`) + `deletion_audit`, before the person drop, inside the same transaction/advisory-lock scope. A post-review concurrency hardening closed a duplicate-retain-record race the gate surfaced. Delivered on PR #1139 (squash-merged to main as `26d7227ea`). A follow-up review round — the autonomous reviewer caught a real DoD gap (a late Claude CHANGES_REQUESTED on #1139): `writeFinancialRecordsForPersonTx` silently `return`ed when no org resolved, so a v2 deletion could complete with zero §6.1 financial_record rows — was fixed fail-closed on PR #1144 (squash-merged as `52a4e3ae5`).

What changed:
- apps/api/src/services/identity-v2/deletion-v2.ts: new `writeFinancialRecordsForPersonTx` helper writes two retain-tier rows per person (tax + chargeback), wired into all five v2 delete paths (`executeDeletionV2`, `deletePersonV2`, `deletePersonIfConsentWithdrawnV2`, `deletePersonIfNoConsentV2`, `deleteArchivedPersonIfStillEligibleV2`) after the consent re-home and before the person DELETE. `financial_record` has no person FK (retain-tier rows outlive the person), so they survive the drop.
- Concurrency hardening (post-review P2 fix): `deletePersonIfNoConsentV2` and `deletePersonV2` lacked the per-person advisory lock, so two concurrent same-person runs could each commit their retain-row side-writes while only one DELETE won — leaving duplicate (4 instead of 2) retain rows. Fix: `acquirePersonLockTx` first + a post-lock `personExistsTx` existence re-check gating the side-writes, mirroring the `deleteArchivedPersonIfStillEligibleV2` / WI-583 pattern; swept across all five paths. `executeDeletionV2` was proven safe via its existing TOCTOU org-claim lock (commented). A code comment near the person DELETE records that subscription teardown (`subscription.payer_person_id ON DELETE RESTRICT`) is a pre-existing CUT-B3/billing gap, out of WI-723 scope.
- apps/api/src/services/identity-v2/consent-v2.integration.test.ts: extended the GREEN deletion test to assert the `financial_record` rows survive the person drop (incl. a non-empty subscription-payload seed) and a new concurrency regression test (two concurrent `deletePersonIfNoConsentV2`, same person + requestedAt) asserting person deleted once, exactly one winner, exactly 2 financial_record + 1 deletion_audit; plus `payload.deletedAt` well-formedness and an `allSettled` fulfillment assertion so a thrown delete cannot masquerade as a clean loser.

No-membership fail-closed (PR #1144): replaced the silent `if (!organizationId) return;` in `writeFinancialRecordsForPersonTx` with a fail-closed throw that aborts the deletion transaction (person not deleted, no partial state; the throw is the Sentry escalation via the Inngest/route boundary — silent recovery is banned in billing-domain code). It rechecks `personExistsTx` first so a benign concurrent-deletion race (a same-org `executeDeletionV2`, which does not take the per-person advisory lock, winning the delete) is an idempotent no-op rather than a spurious throw/retry; the throw fires only for a genuinely orphaned person (still exists, no org resolves).

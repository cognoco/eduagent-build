What was done: Hardened `ThemedMarkdown` (apps/mobile) against LLM-authored-markdown link and image attacks (finding F-027). Link navigation is now scheme-validated and remote images are disabled entirely on this untrusted-markdown path.

What changed: In `apps/mobile/src/components/common/ThemedMarkdown.tsx`, added an exported `isSafeLinkUrl()` guard wired as `onLinkPress` — it allowlists `https:`/`http:` and blocks `javascript:`, `data:`, `file:`, and any unparseable URL by returning `false` (the library suppresses navigation on a `false` return). Set `allowedImageHandlers = []` and `defaultImageHandler = null` so the library's image render rule returns `null` for every image source — no remote image (https tracking pixel, http, or data:) ever loads, closing the zero-click IP-leak vector. Added a local `MarkdownProps` type extension (cast to `ComponentType<MarkdownProps>`) because the library's published `.d.ts` omits `allowedImageHandlers`/`defaultImageHandler`. Tests in `ThemedMarkdown.test.tsx` extend the captured-props type and add scheme-guard cases (blocked: javascript/data/file; allowed: https/http) plus image-disable cases (empty allowlist, null default handler, and a reproduction of the library's image-rule decision proving an https remote image renders nothing).
